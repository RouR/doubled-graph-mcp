# 01 — Intent-интервью

**Роль ИИ:** собрать достаточно информации для заполнения `requirements.xml` и `technology.xml`. Не начинать писать код.

**Правило глубины:** всегда максимальная (см. `methodology/auto-scaling.md`). Никаких «lite»-версий.

---

## Критичное правило — externalize state

Локальные модели со средним контекстом **теряют** ответы из начала интервью к его концу. Поэтому:

- **Задавать вопросы — по 1–3 за раз** (сохраняет фокус в короткой сессии).
- **После каждой батчи** — сохранять полученные ответы в файл `.doubled-graph/drafts/interview.md` append-only. Это persistent state на диске, не в контексте модели.
- **Следующий шаг (`02-requirements.md`) синтезирует `requirements.xml` из этого файла**, а не из истории чата. Это значит — даже если сессия прервалась или контекст вытеснен, данные не потеряны.

`docs/*.xml` при этом **не трогаем** до `doubled-graph init` в шаге 02 (это сохраняется).

### Формат scratchpad'а

```markdown
# Intent Interview — draft

Started: 2026-04-18T10:15Z
Status: in-progress | completed

## Блок A — бизнес-намерение

### Q1. Что делает продукт (одним предложением)?
> <дословный ответ пользователя>

### Q2. Кто пользователь / актор?
> <ответ>

...

## Блок B — стек

### Q7. Предпочтительный язык/стек?
> <ответ>

...

---

Closed: 2026-04-18T10:40Z (when user confirms "всё, можем идти дальше")
```

**Как записывать:** используй Write/Edit tool. Один файл, append после каждого ответа пользователя. Если файл уже существует (например, интервью возобновляется после перерыва) — **не затирай**, добавляй продолжение.

---

## Блок A — бизнес-намерение (4–6 вопросов)

Задавай **по 1–3 за раз**, после каждой батчи — пиши ответы в scratchpad. Примерный порядок:

1. Одним предложением: что делает продукт?
2. Кто пользователь / актор? (можно несколько ролей)
3. Топ-3 действия, которые пользователь совершает в продукте?
4. Критерий успеха — по чему поймёшь, что MVP получился?
5. Нефункциональные ограничения: latency, throughput, regulatory, offline-first? (Обязательно спрашивать — заполняется для всех UC.)
6. (опционально) Интеграции с внешними системами?

Фиксируй ответы в scratchpad **дословно** — они пойдут в `requirements.xml` через промпт 02.

После Блока A — явное подтверждение у пользователя: «Записал эти ответы в `.doubled-graph/drafts/interview.md`. Идём дальше, или что-то уточнить?»

---

## Блок B — технологические предпочтения (3–5 вопросов)

7. Предпочтительный язык/стек продукта? (если «не знаю» — предложи 2 варианта на основе блока A; спроси выбор)
8. Deployment target: локально / Docker / Kubernetes / serverless / edge?
9. Existing constraint на библиотеки (legacy dependency, license requirement)?
10. (опционально) Предпочтения по тестовому фреймворку?
11. (опционально) Ограничения по компьютерным ресурсам (low-memory device и т.п.)?

Про IDE и про модель **не спрашиваем**: методология оптимизирована под локальные модели среднего размера (Qwen 3, DeepSeek V3, GLM-4.5, Llama 4 — см. `methodology/runtime-adapters/local-llm.md`) и работает на любом IDE с поддержкой MCP + tools. Ограничения локальных моделей (компактные промпты, structured decoding) — уже встроены в промпты методологии.

После Блока B — снова append в scratchpad, снова явное подтверждение.

---

## Выход

После интервью:

1. Помечаешь в scratchpad `Status: completed` + `Closed: <ISO-дата>`.
2. **Саммари одной страницей** — зачитываешь пользователю **из scratchpad'а** (не из памяти): продукт, акторы, use-cases (3–5 штук draft), стек-предложение.
3. Подтверждение у пользователя.
4. Переход к шагу 2 — `prompts/new-project/02-requirements.md`. Он должен **явно** начинаться с чтения `.doubled-graph/drafts/interview.md` как источника данных.

---

## Анти-паттерны

- Не начинай писать код / не создавай `docs/` до окончания интервью и `doubled-graph init`.
- Не спорь с выбором пользователя. Если стек кажется неподходящим — фиксируй concern как `NonFunctional` ограничение и идёшь дальше.
- Не собирай все вопросы в один длинный список.
- **Не полагайся на историю чата как на источник ответов.** Локальная модель к концу 13-вопросного интервью забудет начало. Единственный надёжный источник — scratchpad на диске.
- **Не переписывай scratchpad целиком с каждой правкой** — append-only. Правки делай Edit-tool'ом точечно (конкретный блок `### QN`).
- Не удаляй scratchpad после 02-requirements автоматически — пользователь может захотеть вернуться. Удаление — по явной команде или в финальном 07-post-init-sync.
