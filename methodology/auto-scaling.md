# Объём артефактов — всегда максимальная глубина

**Решение 2026-04-18:** методология doubled-graph **не масштабирует глубину артефактов под размер проекта**. Все проекты — от pet-скрипта до enterprise-репозитория — получают максимальный набор полей и разметки.

Ранние черновики обсуждали три профиля (`M`/`S`/`L` — Minimal/Standard/Large) с разной глубиной заполнения `docs/*.xml` и разметки кода. Решение — **отказаться**, использовать только то, что раньше называлось профилем `L`.

---

## Что это значит практически

| Артефакт | Заполнение |
|---|---|
| `docs/requirements.xml` | полный набор `UseCase`, `NonFunctional` для всех UC, traceability-матрица (каждый UC → V-M-*) |
| `docs/technology.xml` | все зависимости, версии, compatibility matrix |
| `docs/development-plan.xml` | модули + фазы + DataFlow + **critical-path** отметки; зависимости по транзитиву; `criticality` заполнен для каждого M-* |
| `docs/knowledge-graph.xml` | все модули + экспорты + CrossLinks + аннотации (`security`/`PII`/`audit`) |
| `docs/verification-plan.xml` | `V-M-*` для каждого модуля с `criticality ∈ {critical, standard}`, gates (pre-commit, pre-merge, pre-prod) |
| `docs/operational-packets.xml` | все форматы заполнены |
| Разметка кода | все 4 уровня якорей: `MODULE_CONTRACT`, `MODULE_MAP`, `CONTRACT: fn`, `BLOCK_*` |
| Блоки в функциях | обязательны в функциях > 20 строк |
| Logs | debug + info + warn + error; `belief`, `correlation_id`, `trace_id`, `anchor` в каждом логе |

Исключение — **`helper` модули** (`criticality="helper"` в `development-plan.xml`): для них блоки и runtime-контракты опциональны (см. `principles.md § 6 — Пропорциональная гранулярность`). Это не профиль проекта, а свойство отдельного модуля.

---

## Почему максимальная глубина везде

**Четыре причины:**

1. **Робастность ценнее экономии.** Опциональность плодит баги: кто-то забыл `criticality`, кто-то не написал `V-M-*`, кто-то пропустил `CrossLinks` — ИИ тиражирует пропуски. Один unambiguous стандарт — ИИ не угадывает, что заполнять.

2. **Инструменты поддерживают только одну форму.** `doubled-graph detect_changes` ищет `critical-path` атрибут и CrossLinks в `development-plan.xml`. Если в "облегчённом" проекте их нет — detect_changes молча недооценивает drift. Альтернатива «адаптировать tooling под профиль» — O(N) сложности на каждый профиль, не окупается.

3. **«Pet-проект, не нужен enterprise-стек»** — анти-паттерн. У pet-проекта тоже есть зависимости между функциями. Чем меньше проект, тем **ниже** overhead от полной разметки — файлов меньше, правил столько же. Экономия «2 минуты на pet-проекте за счёт отказа от CrossLinks» не окупает последующую потерю orientation, когда pet-проект подрос.

4. **ИИ лучше генерирует по полному шаблону.** Локальные модели среднего размера (Qwen 3, DeepSeek V3, …) при частично заполненных шаблонах склонны импровизировать поля. Строгая схема + structured decoding работают стабильнее на полном наборе.

## Где всё же ориентироваться на размер проекта

Не в артефактах — в **выборе multiagent-профиля** (`safe` / `balanced` / `fast`) и в **количестве approval-gates**. См. [approval-checkpoints.md](approval-checkpoints.md).

- `doubled-graph multiagent-execute --profile safe` — для проектов с высокими рисками (payments, PII, регулируемая отрасль), новая команда, критические модули.
- `doubled-graph multiagent-execute --profile balanced` — default, обычный режим.
- `doubled-graph multiagent-execute --profile fast` — mature codebase + сильная verification.

Выбор профиля — на пользователя при запуске, не на основе «размера проекта». Artifact-стек — идентичен во всех случаях.

---

## Что делать, если «я правда хочу меньше артефактов»

Тогда не используй doubled-graph. Частичное применение даёт меньше пользы, чем полный отказ:

- без `V-M-*` — нет привязки кода к verification → ИИ может писать код без тестов → нарушение принципа 9 (прослеживаемость);
- без `CrossLinks` — `detect_changes` не видит дрейф → принцип 7 (код — живой документ) нарушен;
- без полного `criticality` — `impact` не умеет правильно определять risk → правки без guard.

Лёгкие методологии, которые **допустимо** смешивать с самим подходом «ИИ + план», есть, например, BMAD-Method, Kiro-flow, spec-kit. Выбирай одну из них, если doubled-graph кажется overkill. **Не пытайся собрать «doubled-graph Lite» своими руками — это и есть анти-паттерн, упомянутый в причине 1.**
