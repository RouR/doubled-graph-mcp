# Семантически-опорная инженерия кода с двойным графом знаний: методология управляемой разработки с большими языковыми моделями

> Научная статья. Версия 1.0, 2026-04-19. Язык — русский. Статус — препринт.

---

## Аннотация

Массовое внедрение больших языковых моделей (LLM) в инструменты разработки программного обеспечения — Claude Code, Cursor, Windsurf, Continue, Aider — превратило ИИ-ассистента из автокомплита в полноценного участника процесса: модель читает файлы, строит планы, редактирует код, запускает тесты, коммитит. Эта автоматизация, однако, принесла воспроизводимый набор патологий: уверенные галлюцинации, дрейф между документацией и кодом, «улучшения» вне согласованного замысла, потеря контекста в длинных сессиях, вход в циклы исправлений, нарушение инвариантов при рефакторинге. Корень проблемы — несоответствие между архитектурой авторегрессионной языковой модели и требованиями инженерии: модель оптимизирует правдоподобие следующего токена, а не соответствие замыслу; её «память» не является памятью в инженерном смысле; её рассуждение задним числом рационализирует ответ, а не выводит его.

В настоящей работе предлагается методология **семантически-опорной инженерии кода с двойным графом знаний** (далее — методология двойного графа), объединяющая десять взаимоподдерживающих элементов: примат явного замысла, синтез по утверждённому чертежу, семантический каркас кода, контекст через двойной граф, двойное назначение разметки, пропорциональную гранулярность, код как живой документ, наблюдаемое belief-состояние, сквозную прослеживаемость и управляемую автономию. Для каждого элемента разобраны предполагаемый механизм действия, условия применимости и границы эффективности. Элементы классифицированы на *фундаментальные* (удаление которых ломает контур методологии) и *усилители* (удаление снижает эффективность, но сохраняет работоспособность). Настоящая работа — **design/position paper**: она предлагает и обосновывает методологию через аргументацию и ссылки на свойства LLM-архитектур, не через контролируемый эмпирический эксперимент. Валидация на реальных кодовых базах и количественная оценка выигрыша относительно альтернативных практик — открытое направление.

Методология предназначена для применения в локальных open-weights моделях среднего размера (≈30–70 B плотных параметров — например Qwen3-32B, Qwen2.5-72B, Qwen3-Coder-30B, Llama-3.3-70B, GLM-4-32B), развёртываемых в локальной или on-prem инфраструктуре, в IDE с поддержкой внешних инструментов и Model Context Protocol. Отдельное внимание уделено сценариям миграции существующих (brownfield) кодовых баз, для которых исходная концепция «чистой генерации из спецификации» неприменима: предложена политика смены режимов (`migration` ↔ `post_migration`), переключаемая явным полем в конфигурационном файле агента.

**Ключевые слова:** LLM-assisted development, prompt engineering, retrieval-augmented generation, program comprehension, call graph analysis, design by contract, observable AI systems, Chain-of-Verification, log-driven development, brownfield modernization.

## 1. Введение

### 1.1. Проблема

К 2026 году большие языковые модели перестали быть экспериментальной технологией в инженерии ПО. Оценки коммерческого применения показывают: доля кода, генерируемого или редактируемого с прямым участием LLM, превышает половину новых строк в отраслевых репозиториях; агентные системы (Claude Code, Cursor Agent, Windsurf, Devin, Aider) выполняют многошаговые задачи — чтение файлов, построение планов, редактирование, запуск тестов, итеративное исправление — в режиме ограниченной автономии. Однако вместе с ростом автономии обнажились несколько воспроизводимых классов отказов:

1. **Уверенные галлюцинации.** Модель генерирует код или документацию, ссылающуюся на несуществующие функции, неправильные сигнатуры, выдуманные библиотеки. Поверхностно правдоподобный вывод трудно отличить от корректного без автоматической проверки. Работы Ji et al. [1] и Huang et al. [2] документируют структуру этого явления; SimpleQA [3] и TruthfulQA [4] показывают, что даже сильные модели допускают фактические ошибки на банальных вопросах.

2. **Faithfulness-разрыв между рассуждением и ответом.** Исследования Turpin et al. [5] и Lanham et al. [6] демонстрируют, что промежуточная цепочка рассуждений (chain-of-thought) нередко является рационализацией задним числом: ответ фиксируется раньше, чем модель «раскручивает» цепочку.

3. **Деградация точности в длинном контексте.** Liu et al. [7] в работе «Lost in the Middle» показали, что точность извлечения релевантной информации падает, если нужный фрагмент расположен в середине длинной последовательности. Yen et al. [8] расширяют этот вывод на многомерную оценку долгоконтекстных моделей.

4. **Сикофантия и смещения выравнивания.** Sharma et al. [9] задокументировали, что модели систематически меняют ответ под выраженное несогласие пользователя, даже когда исходный ответ верен. Verbosity bias, выявленный в работах по RLHF [10, 11], провоцирует генерацию длинного хеджирующего текста вместо точного.

5. **Дрейф между замыслом и кодом.** В реальной кодовой базе, переживающей десятки коммитов в день, документация, тесты и сам код быстро расходятся. ИИ-ассистент, обученный на гипотезе целостного контекста, не обнаруживает этого рассогласования самостоятельно.

6. **Риск рефакторинга.** Переименование, разделение, удаление символов — операции, требующие знания call graph и неявных зависимостей. LLM, работающая с ограниченным окном, не имеет встроенного представления о blast radius изменения.

Эти отказы — не поломка, а режим работы модели. Они следуют из целевой функции (cross-entropy на следующем токене), а не из дефекта конкретной архитектуры. Следовательно, их нельзя «исправить» выбором модели посильнее: необходима методология, компенсирующая структурные ограничения LLM внешними механизмами — артефактами, проверками, прослеживаемостью, графом.

### 1.2. Цель и вклад

Настоящая работа предлагает методологию, состоящую из десяти элементов, каждый из которых адресует один или несколько перечисленных классов отказов. Вклад работы состоит в следующем:

1. **Интеграция.** Предложена интеграция элементов из разных областей — design by contract [12, 13], graph-based program comprehension [15, 16], retrieval-augmented generation [17], chain-of-verification [18] и прикладной практики log-driven development (сложилась в LLM-observability-инструментарии; единого академического источника не имеет — подробнее в § 11.5; инфраструктурной базой технически выступают distributed tracing [14] и OpenTelemetry [45], но это база *технической реализации*, не источник методики). Аргументирована (на уровне качественных рассуждений и взаимозависимости элементов, не контролируемого эксперимента) гипотеза, что отказ одного элемента компенсируется или частично компенсируется другими.

2. **Анализ каждого элемента.** Для каждого из десяти элементов приведены: предполагаемый механизм действия (почему он, с опорой на опубликованные свойства LLM-архитектур, должен снижать конкретный класс ошибок), условия применимости и границы.

3. **Двойной граф знаний.** Введено разделение между *декларируемым* графом (замысел, утверждённый человеком) и *наблюдаемым* графом (извлечённым из AST и call graph). Предложена процедура их сверки — `detect_changes` — как ядро обнаружения дрейфа. Реализация выполнена как MVP (doubled-graph, Python/Bun, MIT), покрытый базовыми smoke-тестами; production-валидация на реальных кодовых базах — открытое направление.

4. **Политика смены режимов.** Сформулировано переключение `migration` ↔ `post_migration`: в миграции источник истины — код, артефакты догоняют; после миграции — наоборот. Это устраняет искусственное требование «чистой генерации», неприменимое к существующим проектам.

5. **Ориентация на локальные модели среднего размера.** Все элементы спроектированы так, чтобы работать на open-weights моделях ≈30–70 B плотных параметров (например, Qwen3-32B, Qwen2.5-72B, Qwen3-Coder-30B, Llama-3.3-70B, GLM-4-32B), развёрнутых локально или on-prem, с ограниченным контекстом и без frontier-возможностей закрытых API.

**Характер работы.** Настоящая работа — **design / position paper**, не empirical study. Все утверждения о «почему элемент работает» — аргументы правдоподобия, опирающиеся на (а) опубликованные свойства LLM-архитектур, (б) устоявшиеся практики инженерии ПО (design by contract, structured logging, graph-based comprehension, RAG), (в) дизайнерские соображения об интеграции этих практик. Утверждения формата «выигрывает в X раз», «снижает инциденты на Y%», «хорошо работает на задачах класса Z» в работе *не делаются* — для них требуется контролируемая оценка, отсутствующая в текущем состоянии работы. Любое наблюдение из индустриальной практики помечено как таковое (не «доказано»).

## 2. Теоретический фундамент

### 2.1. Авторегрессия и природа «знаний» в LLM

LLM — авторегрессионная модель, минимизирующая cross-entropy loss на предсказании следующего токена над текстовым корпусом. Это имеет несколько прямых следствий, определяющих инженерные границы:

1. **Правдоподобие ≠ истинность.** Целевая функция оптимизирует статистическую вероятность последовательности, а не её соответствие миру. Модель «не знает», верен ли факт; она знает, что токен `Париж` статистически следует за «Столица Франции — ». Этот разрыв между правдоподобием и истинностью есть структурная причина галлюцинаций (Ji et al. [1]).

2. **Один прямой проход через сеть имеет фиксированную последовательную глубину и ограниченную выразительную мощность.** Прямой проход (forward pass) — одно применение модели к входу: вход проходит через фиксированное число трансформер-слоёв, и на выходе получается распределение вероятностей следующего токена. Число слоёв задаётся архитектурой модели. Для крупных открытых моделей (проверено по `num_hidden_layers` в соответствующих `config.json` на HuggingFace) диапазон широк: 48 у Llama 4 Scout/Maverick (MoE с «широкой-мелкой» конфигурацией), 56 у Mixtral 8x22B, 61 у DeepSeek V3/R1, 80 у Llama 3/3.1/3.3 70B и Qwen 2.5 72B, 88 у Mistral Large 2, 92 у GLM-4.5, 94 у Qwen3 235B-A22B, 126 у Llama 3.1 405B. Это число слоёв не следует путать с полным объёмом вычислений на токен (он зависит от ширины скрытого состояния, размера FFN-блоков, числа голов внимания, а для MoE — от числа активных экспертов) и **не является прямой мерой «шагов рассуждения»**. Тем не менее существует нижняя оценка выразительной мощности трансформера в рамках одного прямого прохода: без возможности записать промежуточные результаты обратно в контекст, модель ограничена классом функций, вычислимых за это число слоёв.

   Преодоление этого ограничения в LLM происходит **через авторегрессию**, а не через увеличение глубины. Генерация токен-за-токеном эквивалентна возможности писать промежуточные результаты в контекст и читать их на следующих шагах — именно так chain-of-thought, внешние инструменты и декомпозиция задач расширяют фактический класс вычислимого. Поэтому:

   - **Chain-of-thought** внутри одной сессии помогает, так как модель генерирует и затем *перечитывает* промежуточные шаги, используя контекст как scratchpad.
   - **Внешняя декомпозиция на несколько вызовов** (Wei et al. [19], Kojima et al. [20], Wang et al. [21]) даёт дополнительно: чистый контекст для каждого шага (нет интерференции между ролями), возможность валидировать выход шага *до* передачи дальше (см. § 2.5 про CoVe), и независимость от длины промежуточных рассуждений.

   Для инженерии ПО это практически означает: длинные многошаговые задачи (построение плана → генерация модуля → тестирование → исправление) разумно исполнять *серией вызовов с материализацией промежуточных артефактов на диск*, а не одним gigantic-промптом — не столько из-за «нехватки слоёв», сколько из-за а) накопления ошибок в одной авторегрессионной траектории, б) потери способности локально валидировать, в) деградации точности в длинном контексте.

3. **Внимание — гетерогенный механизм воспроизведения.** Механистическая интерпретация показала, что в обученных трансформерах существуют специализированные **головы копирования** (в оригинальной работе — «induction heads», Elhage et al. [46]). Они реализуют внутри модели следующий алгоритм: когда при генерации встречается токен X, голова ищет в контексте предыдущие вхождения X и «протягивает» вперёд тот токен, который шёл после X раньше. Иначе говоря, если в контексте уже было «X → Y», а сейчас снова встретился X, голова предсказывает Y. Этот механизм объясняет, почему современные LLM иногда *удивительно точно* воспроизводят длинные идентификаторы, пути, сигнатуры и заученные из нескольких примеров паттерны (in-context learning).

   Однако копирование — не гарантия. Головы копирования могут уступать «мягким» паттернам внимания, выдающим правдоподобно искажённую версию, когда:
   - (а) копируемая последовательность из контекста *статистически реже*, чем похожая пара токенов, часто встречавшаяся в обучающих данных — параметрическая память «перебивает» копирование (например, редкое имя функции заменяется на похожее, но более частотное);
   - (б) источник и точка воспроизведения разделены длинным контекстом, и деградация долгого контекста (см. § 2.3) сказывается уже на самой операции копирования.

   Практический вывод: не следует полагаться на точное воспроизведение неочевидных строк — имён функций, точных хешей, длинных идентификаторов — без внешней проверки (линтер, цикл «запустить компилятор и прочитать ошибку», отдельный проверяющий агент). Копирование статистически частое, но не всегда верное.

4. **Контекст — рабочая память, не долговременная.** Состояние, не попавшее в contextwindow следующего вызова, теряется. Для многошаговых процессов это требует явной экстернализации состояния — через файлы, структурированные объекты, scratchpad'ы.

### 2.2. Alignment и смещения

После pre-training модели дообучаются на предпочтениях людей — через RLHF (Christiano et al. [22], Ouyang et al. [10]) или более новые методы (DPO — Rafailov et al. [23]; KTO; ORPO). Alignment оптимизирует предпочтения оценщиков, а не инженерную полезность. Это порождает систематические смещения:

- **Verbosity bias** — предпочтение длинных ответов, даже если они менее точны [11].
- **Sycophancy** — стремление согласиться с пользователем, включая случаи, когда пользователь не прав [9].
- **Хеджирование** — избыточное количество оговорок, маскирующих низкую уверенность.

Эти смещения особенно опасны в диалогах с итеративной правкой: каждая итерация увеличивает риск deferring-to-user (модель постепенно уступает пользователю), а итоговый вывод может не соответствовать начальному архитектурному решению.

### 2.3. Длинный контекст и его пределы

Расширение контекстного окна до 128K, 200K и более токенов создаёт иллюзию решения проблемы памяти. Эмпирические работы показывают, что это лишь частично верно:

- **Lost in the Middle** (Liu et al. [7]): точность извлечения релевантных фактов из длинного контекста деградирует для документов, расположенных в середине окна.
- **HELMET** (Yen et al. [8]): более многомерная оценка долгоконтекстных моделей подтверждает, что они не «читают» весь контекст равномерно; поведение зависит от задачи.
- **Контекст — дорогой ресурс.** Time и cost масштабируются линейно (prefill) и квадратично (внимание) с длиной контекста.

Следовательно, «положить весь репозиторий в контекст» не является инженерным решением — даже при наличии модели на 1M токенов. Для крупных кодовых баз необходимо **селективное извлечение** (retrieval) — механизм, подающий в контекст модели только релевантные задаче фрагменты кода (нужные функции, их зависимости, соответствующие фрагменты документации), а не весь корпус. В настоящей работе такое извлечение опирается на граф-структуру кода: по запрошенному символу граф возвращает окружение — прямых вызывающих, зависимости, связанные артефакты.

### 2.4. Retrieval-Augmented Generation и её пределы

RAG (Lewis et al. [17]; Gao et al. [24]) выносит факты во внешний индекс и возвращает их через извлечение. Это снижает частоту галлюцинаций на фактуальных задачах — при условии, что извлечение приносит релевантные источники. Однако:

- **Простая векторная RAG** (чанкинг + cosine similarity) плохо справляется с многошаговыми запросами и с запросами, требующими глобального взгляда на корпус.
- **GraphRAG** (Edge et al. [25]) и **RAPTOR** (Sarthi et al. [26]) расширяют наивную RAG графовой и иерархической структурой, что существенно улучшает качество на мультихоп-запросах и глобальных summarization-задачах.
- **Код — особый случай.** Для кода семантическая близость чанков недостаточна: нужны точные структурные отношения (вызовы, импорты, наследование, использование типов), извлекаемые из AST.

Методология двойного графа, предлагаемая в настоящей работе, явно опирается на эту второе ветку: **declared graph** кодирует замысел, **observed graph** извлекается инструментами статического анализа (tree-sitter, AST-парсеры) из самого кода.

### 2.5. Chain-of-Verification и антигаллюцинационный контур

Dhuliawala et al. [18] предложили Chain-of-Verification (CoVe) — четырёхшаговый протокол: черновик, вопросы к черновику, ответы на вопросы в изолированном контексте, финальный ответ с учётом проверок. Независимость ответов на проверочные вопросы принципиальна: она устраняет confirmation bias, наблюдаемый, когда тот же контекст используется и для генерации, и для проверки.

Для кода CoVe-аналог реализуется как связка `generator → verifier → tests`, где:
- generator — основной промпт (редактирование, написание функции);
- verifier — отдельный вызов с низкой температурой, проверяющий соответствие контракту и инвариантам;
- tests — исполняемая проверка, не зависящая от LLM.

### 2.6. Design by Contract и программные якоря

Методология Meyer [12] предложила формулировать предусловия, постусловия и инварианты как часть интерфейса. В современных языках это реализуется через assert'ы, Pydantic/Zod-валидацию на границах, runtime-контракты (`@beartype`, `icontract`, `deal`, `contracts`-crate). В AI-контексте контракты выполняют дополнительную функцию: они служат **семантическим якорем** для модели. Явно заданные pre/post-conditions в docstring или структурированном блоке уменьшают множество правдоподобных реализаций до множества правильных.

Related research on program comprehension показывает, что имена идентификаторов и docstring'и являются первичными носителями семантики в glass-box-моделях понимания кода (Hindle et al. [27]; Allamanis et al. [28]).

### 2.7. Log-Driven Development и наблюдаемость

В LLM-инженерии сосуществуют два разных слоя наблюдаемости, которые важно различать.

**Слой 1 — observability ассистента (LLM-вызовов).** Инструменты типа LangSmith, Arize Phoenix, W&B Weave, OpenLLMetry, Langfuse записывают каждый вызов LLM: промпт, ответ, метрики (latency, токены, стоимость), валидацию (schema pass/fail). Это отвечает на вопрос «почему ассистент в такой-то сессии предложил такое-то изменение?». Это — трассировка *инференс-процесса*.

**Слой 2 — observability приложения с зашитыми допущениями.** В настоящей работе этот слой называется *log-driven development с belief-полем*: код, сгенерированный ассистентом, содержит структурированные лог-события, в которых зафиксированы *runtime-допущения автора функции* — какое состояние считалось входом, какие инварианты предполагались в точке исполнения. Это отвечает на вопрос «какое допущение было зашито в код и сохранилось ли оно в конкретном прогоне?». Это — трассировка *допущений, материализованных в исходнике*, не инференс-процесса.

Оба слоя нужны и *не заменяют друг друга*. Слой 1 объясняет поведение ассистента при генерации; слой 2 — поведение сгенерированного кода в продакшене. В методологии, описываемой ниже, слой 2 является опорным; слой 1 рекомендуется дополнительно, но не включён в базовый набор.

Методология предполагает три канонических сценария:

- **Новый проект.** Замысел оформляется первым; код генерируется из утверждённых артефактов. Режим `post_migration` с первого шага.
- **Миграция существующего (brownfield).** Код существует, замысел не формализован. Режим `migration`: код — источник истины, артефакты восстанавливаются; по завершении — переход в `post_migration`.
- **Поддержка (день-в-день).** Итеративные правки, интерпретируемые в текущем режиме. Дрейф обнаруживается автоматически и разрешается по политике.

Центральным артефактом согласованности является **двойной граф**: declared + observed. Declared пишется человеком и/или ИИ по замыслу и представляет, что *должно* быть. Observed извлекается из AST + call graph и представляет, что *есть*. Их сверка — источник сигналов о дрейфе.

В разделах 4–13 каждый элемент рассмотрен в одной и той же схеме:
- **Описание.** Что именно делает элемент.
- **Механизм действия.** Как он снижает конкретный класс ошибок.
- **Условия применимости.** Когда выигрыш окупает издержки.
- **Границы и провалы.** Когда применение либо бесполезно, либо вредно.
- **Связанные работы.**

## 4. Элемент 1. Примат явного замысла

### 4.1. Описание

Любой код — следствие утверждённого артефакта замысла. В новой разработке это означает цепочку: бизнес-цель → функциональные требования в нотации Actor–Action–Goal → выбор стека → план модулей с контрактами → верификационный план → и только после всего этого — кодогенерация. Каждый артефакт материализован в структурированной форме (XML, пригодной для автоматической валидации) и проходит одобрение человеком.

В миграции замысел восстанавливается из существующего кода через reverse-engineering: AST-анализ, чтение README, допрос оставленной в репозитории документации, интервью с пользователем. Но и в миграции требование *одобряется человеком* до того, как начнётся новая работа.

### 4.2. Механизм действия

Элемент адресует одновременно несколько отказов:

1. **Галлюцинации «вне задачи».** Модель без явного замысла склонна генерировать правдоподобный, но не соответствующий намерению код. Явный артефакт локализует пространство допустимых решений.

2. **Post-hoc rationalization.** Turpin et al. [5] показали, что цепочка рассуждений может не отражать реальный вывод. Явный замысел фиксирует намерение до генерации, превращая промежуточные шаги в проверку соответствия, а не в оправдание ответа.

3. **Сикофантия.** Sharma et al. [9]: модель меняет ответ под давление пользователя. Артефакт, утверждённый до диалога, выступает якорем: агент может ссылаться на него при разногласии.

4. **Разрыв замысла и исполнения при смене сессий.** Между сессиями контекст теряется, но артефакт на диске — не теряется.

### 4.3. Условия применимости

Элемент окупается, когда:
- Задача выходит за пределы ≤ 30 строк кода или одного рефакторинга.
- Код будет поддерживаться (а не сгенерирован для одноразового скрипта).
- В проекте более одного разработчика (или один, но планирует возвращаться).
- Ожидается использование ИИ-ассистентов с разной памятью (смена IDE, модели).

### 4.4. Границы и провалы

1. **Over-specification на старте.** Попытка заранее специфицировать всё детализированно парализует стартап-динамику и провоцирует «бесконечное планирование», в котором каждая итерация стирает конкретику предыдущей. Стандартная индустриальная рекомендация — условное «правило 80%»: переходить к исполнению, когда ясно около 80% замысла, оставляя часть деталей на проявление при реализации.

   Это правило **применимо с оговорками**. Оно полезно как защита от paralysis by analysis, но несёт собственные риски:
   - **Замысел не проявляется сам по себе — он требует активного discovery.** Оставшиеся 20% *могут* проявиться при реализации, только если инженер активно ищет их. В пассивном режиме «пишу код и смотрю, что всплывёт» они не проявляются — всплывают как инциденты в продакшене.
   - **Асимметричная стоимость.** 20% незафиксированного замысла в критическом модуле (auth, payment, data-integrity) могут потребовать переписать 80% реализации. Экономия на планировании здесь обманчива.
   - **Иллюзия прогресса.** Психологически «я уже на 80%» ощущается как «можно начинать» — это известный когнитивный bias, который срабатывает даже когда реальная ясность ниже заявленной.

   **Honest применение правила:**
   - **Некритические пути** (`criticality ∈ {standard, helper}`): правило применимо, издержки пропущенных деталей ограничены, реализация действительно помогает заземлить оставшиеся решения.
   - **Критические пути** (`criticality = critical`): правило *не применяется*. Для модулей с безопасностью, деньгами, PII, compliance или безотказностью замысел должен быть зафиксирован подробно *до* реализации; оставшиеся неясности явно фиксируются как `OpenQuestions` в `requirements.xml` или как записи в `DRIFT.md`, не как молчаливые предположения.
   - На уровне проекта в целом: правило 80% относится к *составу* плана, не к *качеству* его критических частей. Можно стартовать с частично-детализированным планом, если критические модули в нём специфицированы полно, а standard/helper — в основных чертах.

2. **Replacement of judgment.** Артефакт не должен заменять решение инженера; он его *фиксирует*. Если документ написан «потому что методология требует», он сам становится мусором и источником drift.

3. **Brownfield-парадокс.** В существующих проектах замысел первичен *исторически*, но артефакта часто нет. Попытка начать с нуля, проигнорировав существующий код, приводит к generic architecture, несвязанной с реальностью. Этот паттерн неоднократно описан в отчётах о внедрении BMAD-методологии и смежных brownfield-подходов в prompt-engineering сообществах; количественных данных о частоте встречаемости автору неизвестно. Митигация — режим `migration` (§ 15.3 позиционирует BMAD как комплементарную методологию, не failure mode; failure mode — конкретно generic-architecture-without-discovery).

4. **Оптовое переписывание.** ИИ, ловко следующий замыслу, склонен переписать окрестный код «для согласованности». Это нарушает границу scope. Митигация — жёсткие правила в режиме `migration`: «правь только то, что в scope волны».

### 4.5. Связанные работы

Декомпозиция как защитная техника — Wei et al. [19], Kojima et al. [20], Wang et al. [21] (Plan-and-Solve), Khot et al. [29] (Decomposed Prompting), Zhou et al. [30] (Least-to-Most). Формализация требований в виде EARS-конвенции (Mavin et al. [31]) полезна для reverse-engineering. Работа Bertrand Meyer [12] даёт фундамент для перевода намерения в контракты.

## 5. Элемент 2. Синтез по утверждённому чертежу

### 5.1. Описание

Каждый артефакт одобряется человеком перед тем, как становится входом для следующего шага. Одобрение — не формальность, а активная проверка ограниченного объёма изменений. На практике это реализуется через набор approval-точек в skill-командах ИИ-ассистента: после декомпозиции модулей, перед execution queue, в многоагентных волнах, при изменении контракта, перед рефакторингом, при отладке.

В текущей реализации методологии существуют три профиля исполнения для многоагентных сессий:
- `safe` — approval перед каждой волной;
- `balanced` — один up-front approval и отчёт по завершении;
- `fast` — минимум approval'ов, максимум автономии.

### 5.2. Механизм действия

1. **Прерывает каскад ошибок.** В многошаговом LLM-прогоне ошибочный вывод шага N становится входом шага N+1, и последующие шаги не обнаруживают ошибочную посылку, а рационализируют её: авторегрессия и каузальное чтение (модель не «возвращается» к предыдущим токенам, чтобы их пересмотреть) означают, что ошибки *коррелированы* во времени, а не независимы. Gates разрывают эту корреляцию: каждый шаг проверяется до того, как его выход станет контекстом следующего, — и ошибка локализуется в одном шаге вместо распространения через весь прогон.

   *Что здесь сознательно не делается — и почему.* Напрашивается мотивирующая арифметика вида «0.9¹⁰ ≈ 0.35, т.е. при 10%-ой ошибке на шаг полностью корректными остаются ≈ треть десятишаговых прогонов». Этот расчёт предполагает *независимость* ошибок между шагами — ровно то предположение, которое в LLM-пайплайнах нарушено по приведённым выше причинам; 0.35 в нём — *верхняя* граница, не средняя, а реальная вероятность корректного прогона ниже. Приводить такую цифру как иллюстрацию — плохая риторика: якорный эффект оставляет у читателя «треть корректна», а не оговорку «в реальности ниже», даже когда оговорка явно произнесена. Поэтому числовая иллюстрация в работе намеренно не используется; аргумент в пользу gates держится на качественном утверждении о корреляции ошибок, а не на идеализированной модели независимости.

2. **Нейтрализует сикофантию.** Одобрение артефакта фиксирует позицию до диалога. В последующих итерациях модель может сослаться на утверждённый артефакт, а не уступать текущему настроению пользователя.

3. **Создаёт точки rollback.** Если ошибка обнаружена на шаге 5, возврат к шагу 3 означает откат к утверждённому состоянию, а не реконструкцию всего контекста.

### 5.3. Условия применимости

Approval-gates окупаются, когда стоимость ошибки на последующих шагах превышает cost одобрения. Для задач с высокой стоимостью ошибки (payment, auth, data integrity) — всегда. Для низкоценных (мелкие правки, типографика) — избыточно.

### 5.4. Границы и провалы

1. **Approval fatigue.** Если gates расставлены слишком часто, пользователь начинает одобрять «не глядя». Методология исходит из принципа: gate там, где агент не может продолжить без человеческого решения, и только там.

2. **Bottleneck человеческого внимания.** При активной разработке с многими волнами одобрение задерживает работу. Частичное решение — асинхронный flow: агент подготавливает батч решений, человек одобряет пакетом.

3. **Зловредный учёт.** Agent может представить решение так, что одобрение кажется техническим, хотя содержит архитектурное изменение. Митигация — структура представления, требующая явно выделять contract changes.

### 5.5. Связанные работы

Концепция human-in-the-loop восходит к Amershi et al. [32]. Плановая-исполнение декомпозиция формализована в LangGraph, CrewAI, AutoGen. Constitutional AI (Bai et al. [33]) встраивает самоаудит в модель, но не заменяет явного одобрения для инженерных контрактов.

## 6. Элемент 3. Семантический каркас кода

### 6.1. Описание

Код содержит **парные якоря** в комментариях:

- На уровне файла: `START_MODULE_CONTRACT` / `END_MODULE_CONTRACT`, `START_MODULE_MAP` / `END_MODULE_MAP`, `START_CHANGE_SUMMARY` / `END_CHANGE_SUMMARY`.
- На уровне функции: `START_CONTRACT: <name>` / `END_CONTRACT: <name>`.
- На уровне блока внутри функции: `START_BLOCK_<NAME>` / `END_BLOCK_<NAME>`.

Имена якорей привязаны к идентификаторам из плана разработки: `M-<MODULE>`, `UC-<USECASE>`, `V-M-<VERIFICATION>`. Синтаксис комментария зависит от языка (для Python — `#`, для TypeScript/Java/Go/Rust — `//`).

### 6.2. Механизм действия

1. **Разделение ролей кусков контекста — композиция трёх peer-reviewed механизмов.** Парные теги и системные разделители давно применяются в промпт-инженерии как способ отличить инструкции, данные и примеры (рекомендации Anthropic [35]; руководство OpenAI; DSPy-сигнатуры — Khattab et al. [39]). Прямых рецензируемых исследований формата «XML-теги в промптах → измеряемое перераспределение attention» автору не известно, и такое измерение в настоящей работе *не* заявлено. Ранее цитировавшаяся в черновике работа Hewitt & Manning [34] относится к probing-исследованиям синтаксических структур в эмбеддингах и к данному классу явлений неприменима — ссылка на неё в обоснование тезиса снята. Вместо «правдоподобного объяснения» механизм действия собирается из трёх отдельно опубликованных результатов:

   (a) **Induction heads как матчеры уникальных парных последовательностей (Elhage et al. [46]).** Механистическая интерпретация показала: в обученных трансформерах существуют специализированные головы копирования, реализующие правило «если ранее в контексте было X→Y, и сейчас снова встретился X — предсказывай Y» (см. § 2.1, п. 3). Парные теги `<M-AUTH>…</M-AUTH>` и якоря `START_BLOCK_X…END_BLOCK_X` — идеальный вход для этого класса голов: открывающая и закрывающая метка образуют редкую, уникальную, идентичную с точностью до префикса пару токенов, которую induction head надёжно связывает независимо от расстояния между ними. Это не измерение на задаче XML-тегов, но это peer-reviewed описание того самого класса внимания, на работу которого такие теги рассчитаны; утверждение «теги удобны для induction heads» — не метафора, а прямое следствие определения механизма.

   (b) **Материализация скрытого плана в контекст (Wei et al. [19]; Wang et al. [21]; Khattab et al. [39]).** По аргументу § 2.1 п. 2, авторегрессия эквивалентна возможности писать промежуточные результаты в контекст и читать их на следующих шагах; на этом основана вся CoT-линия и DSPy-signatures. Разметка выносит структуру решения из hidden state в видимый контекст: модели не нужно одновременно удерживать «что я строю» и «как я это строю» — она последовательно заполняет именованные блоки. Это не специфично для XML-тегов, но XML-теги — один из наиболее чистых синтаксических носителей такого плана.

   (c) **Стабильные семантические координаты вместо номеров строк.** Позиционные кодировки трансформера закладывают позицию токена через суперпозицию гармоник разных частот: модель «видит» текст не как плоско нумерованную последовательность, а как вложенную структуру разномасштабных позиций. Из-за этого плоские номера строк — нестабильная адресация (любая правка сдвигает их); именованный якорь `START_BLOCK_DECODE_JWT`, напротив, даёт уникальный, устойчивый к локальным правкам адрес, к которому можно привязать запрос к фрагменту или патч (эмпирически этот класс якорей и предлагают современные prompting-практики Anthropic/OpenAI для RAG-редактирования кода).

   *Важная оговорка.* Каждый из (a)–(c) опубликован и peer-reviewed в своём домене; но их совместное применение именно к XML-тегам и парным якорям в коде — композиционный аргумент настоящей работы, а не прямое измерение. Сильного утверждения формата «теги дают +X% accuracy на задаче Y» в работе *не* делается — утверждается только, что предполагаемый механизм собирается из опубликованных компонент, а не из фольклора индустрии. Количественное измерение эффекта именно XML-разметки на attention — открытое направление.

2. **Устранение семантической интерференции.** Без разметки модель вынуждена сама определять, какой кусок кода — контракт, какой — тело, какой — пример. С разметкой эти роли даны явно.

3. **Zero-context survival.** Файл, открытый впервые, должен быть понятен агенту без видимости других файлов. Якоря + docstring обеспечивают это: контракт виден до тела функции.

4. **Поддержка селективной навигации.** При использовании **code-aware сплиттера, осведомлённого о конвенции якорей**, их границы служат естественными точками разреза чанков, сохраняющими семантическую целостность блока. Стандартные сплиттеры — фиксированного размера, `RecursiveCharacterTextSplitter` (LangChain, LlamaIndex), разбиения по предложениям/абзацам — конвенцию не знают и якоря игнорируют. Даже AST-ориентированные сплиттеры (LlamaIndex `CodeSplitter`, tree-sitter-based индексаторы в Cursor/Continue, Aider repomap) режут по синтаксическим границам языка (функция, класс), а не по кастомным маркерам в комментариях. Следовательно, ценность якорей для извлечения реализуется не через «любой RAG автоматически», а через специализированный инструмент навигации (в нашей реализации — `file_show --contracts --blocks`) или через собственный сплиттер, написанный под конвенцию. Без такого слоя якоря остаются полезны для генерации (пункты 1–3), но не для RAG-индексации.

### 6.3. Условия применимости

Семантический каркас окупается во всех кодовых базах, предполагающих взаимодействие с LLM: от генерации до ревью. Издержки — порядка 10–20 строк комментариев на функцию средней сложности. Выгода — существенное снижение ошибок генерации на зависимых функциях и ускорение навигации в длинных файлах.

### 6.4. Границы и провалы

1. **Разметка ради разметки.** Если якоря ставятся механически без осмысленных LINKS (на несуществующие UC, случайные M-*), они превращаются в шум и увеличивают cognitive load без выгоды.

2. **Языки без docstring-конвенции.** В C разметка через comments остаётся валидной, но менее органичной; в HTML/CSS — якоря становятся мусором. Методология применяет адаптеры к языкам (Python: PEP 257 docstring + `#` якоря; TS: JSDoc + `//`; Rust: `///` + `//`; Go: godoc + `//`).

3. **Ручная синхронизация имён.** Переименование функции без обновления `START_CONTRACT: <new_name>` и ссылки в LINKS приводит к рассогласованию. Митигация — автоматические rename-инструменты, проверка `doubled-graph lint`.

4. **Чрезмерная вложенность.** Якоря в блоках < 5 строк — лишний шум. Настоящая методология задаёт в качестве рекомендуемого размера блока ≤ 500 токенов (≈ 30–50 строк кода) — это дизайн-эвристика, основанная на размерах, с которыми комфортно работают современные LLM-инструменты при чтении фрагмента; строгого обоснования числа нет.

### 6.5. Связанные работы

- **Design by Contract** — Meyer [12]; расширения на ОО-языки — Leavens & Cheon [13].
- **Literate programming** — Knuth [36]. Якоря — минималистский преемник idea Knuth в контексте LLM.
- **Tree-sitter** и подобные AST-парсеры — Brand et al. [37], Bradley et al. [38] — позволяют извлекать семантические рамки автоматически. Явные якоря в исходнике упрощают эту задачу и делают её работающей даже для форматов, которые плохо парсятся AST.
- **Prompt architecture** — обширная практика использования XML/Markdown-тегов для структурирования промптов (рекомендации Anthropic [35]; Khattab et al. [39] — DSPy).

## 7. Элемент 4. Контекст через двойной граф знаний

### 7.1. Описание

Методология опирается на **два независимых графа**:

1. **Declared graph** — написанный ИИ/человеком в `docs/knowledge-graph.xml`. Представляет замысел: какие модули есть, что они экспортируют, какие имеют CrossLinks, какие аннотации (например, `security: PII-touching`).

2. **Observed graph** — вычисляемый из реального кода AST-парсером (tree-sitter, LSP-серверы, AST-библиотеки языка). Представляет факт: реальные определения функций, реальные вызовы, реальные импорты.

Их сверка — операция `detect_changes` — даёт структурированный отчёт:

- `code_without_module` — файлы, существующие в `Paths` несуществующих модулей (или не входящие ни в один модуль);
- `module_without_code` — модули, описанные в плане, но не имеющие соответствующего кода;
- `contract_mismatch` — экспортов в declared нет, а в observed есть (или наоборот);
- `stale_crosslinks` — ссылки на несуществующие модули;
- `missing_verification` — критические модули без `V-M-*` записей;
- `markup_missing` — файлы без якорей модуль-контракта.

### 7.2. Механизм действия

1. **Дизайн-гипотеза: селективное извлечение через граф должно ослаблять воздействие lost in the middle.** Здесь важно явно развести две пропозиции, чтобы не переносить достоверность одной на другую.

   *Установленный факт (цитируется).* Liu et al. [7] показали: точность извлечения релевантного фрагмента из длинного окна деградирует, если фрагмент расположен в его середине. Работа описывает феномен применительно к задаче чтения документов; механизм объяснения фиксированный, количественные кривые — в [7].

   *Гипотеза настоящей работы (не следует из [7]).* Подача агенту условно 2K токенов, отобранных через граф зависимостей (интересующие модули + прямые связи), вместо 500K токенов всего репозитория — должна уменьшать долю случаев, в которых релевантный код оказывается в «провальной» середине окна. Это рассуждение — качественное, правдоподобное следствие из (а) факта Liu et al., (б) архитектурного выбора двойного графа как источника извлечения. Оно не является эмпирическим результатом: контролируемый замер accuracy/recall на парах «репозиторий × задача» с графом и без в настоящей работе не проводился. Количественная валидация — открытое направление (см. § 15 об ограничениях и § 16 о валидации).

   *Оговорки, вытекающие из самой гипотезы.* (а) Граф — *усилитель* качества извлечения, а не решение проблемы: внутри выбранных 2K токенов «потеря в середине» по-прежнему может проявляться на границах функций; (б) качество извлечения пропорционально качеству declared graph — устаревший или неточный declared даёт уверенно неверный результат (см. § 7.4, п. 5); (в) граф видит только *зафиксированные* связи (CrossLinks, импорты, вызовы), а неявные зависимости через разделяемое изменяемое состояние (синглтоны, глобальные переменные, БД, шина событий, файловая система, сетевые побочные эффекты) остаются невидимыми (см. § 7.4, п. 6); (г) выигрыш от «короткого окна» может быть частично съеден худшим качеством ретривера, если граф построен по устаревшему или неполному коду.

2. **Разделяет «что хотели» и «что получилось».** В классической RAG ответ строится на найденных чанках без явной модели намерения. Двойной граф делает это разделение первичным.

3. **Ранняя детекция дрейфа.** Любое изменение кода, не зафиксированное в declared, немедленно проявляется как элемент `contract_mismatch` или `code_without_module`.

4. **Рефакторинг с знанием blast radius.** Перед изменением функции агент запрашивает `impact(target=<symbol>, direction=upstream)` — получает depth-1 (прямые вызывающие, WILL_BREAK), depth-2 (косвенные, LIKELY_AFFECTED), depth-3 (транзитивные, MAY_NEED_TESTING). Это конкретное число, а не догадка.

### 7.3. Условия применимости

- Кодовая база > 1 модуля (для пет-скриптов — избыточно).
- Язык, поддерживаемый AST-парсером (Python, TypeScript, Java, Go, Rust, C#, Swift — через адаптеры; Haskell — через tree-sitter с дополнительным парсингом).
- Наличие инструментария статического анализа.

### 7.4. Границы и провалы

1. **AST не видит динамики.** Eval, `__getattr__`, метапрограммирование, dynamic imports — теневые зоны для observed graph. Митигация — ручная аннотация таких точек в MODULE_CONTRACT.

2. **Индекс может устареть.** Между коммитом и пересчётом observed graph существует окно, в котором граф стал stale. Митигация — git post-commit hook, запускающий `analyze` автоматически.

3. **Размер графа для очень больших монорепо.** Для репо с > 10⁶ символов наивная визуализация и полный хранение графа в памяти становятся проблемой. Митигация — инкрементальный `analyze` по изменённым файлам; графовые СУБД (FalkorDB, Neo4j, Kùzu) как бэкенд.

4. **Declared/observed drift сам по себе не ошибка.** Он — сигнал. Интерпретация — в политике дрейфа (раздел 13).

5. **Качество извлечения зависит от качества declared graph.** Если declared устарел — агент получает «уверенный неверный ответ»: фрагменты из модулей, которые считались релевантными, но больше таковыми не являются. Формально это попадает под п. 4 как дрейф, но оперативно для извлечения это выглядит иначе — *ответ уже дан и использован*, а сигнал дрейфа придёт только на следующем коммите/refresh.

   Что уже реализовано в текущей версии методологии: **`refresh --scope targeted`** как команда ([cli.py:483](../doubled-graph/src/doubled_graph/cli.py), [server.py:131](../doubled-graph/src/doubled_graph/server.py)), вызываемая автоматически *после* правок кода — в `on-after-edit.md`, `on-drift-detected.md`, `on-user-manual-edit.md`, `04-markup-codebase.md`, `06-generate.md`. Вызовы declared-графа после этих прохождений остаются актуальными.

   Чего **нет** в текущей реализации и что оставлено для дальнейшей работы:
   - принудительный refresh *перед* критическими операциями (рефакторинг, impact на высокорисковый символ) — сейчас это proposed-практика, не встроенная в промпты;
   - автоматический мониторинг доли stale-узлов declared-графа как метрика здоровья репозитория;
   - пересечение результатов извлечения с observed graph на поздних этапах сессии (используется вручную через `detect_changes`, но не встроено в retrieve-путь).

   До их реализации дисциплина сверки лежит на пользователе и на промпт-конвенции.

6. **Граф видит только зафиксированные связи.** CrossLinks, импорты, прямые вызовы, явное наследование — распознаются. Неявные зависимости — нет: shared mutable state (глобальные переменные, синглтоны), взаимодействие через БД (две функции пишут/читают одну таблицу, но call-graph их не связывает), event bus / pub-sub, side effects через файловую систему или сеть, dependency injection через runtime-lookup. Для кода, активно использующего такие паттерны, blast-radius по графу — *нижняя* оценка реального влияния. Митигация: (а) явная аннотация shared-state в MODULE_CONTRACT (`// shared-state: global.session`, `// side-effect: file-write /tmp/...`); (б) integration-тесты, покрывающие цепочки через shared state; (в) для данных — формализация «owner-модуль таблицы» как конвенция, позволяющая построить виртуальные связи.

### 7.5. Связанные работы

- **Graph-based program comprehension** — обзорные работы Binkley et al. [15], Storey [16].
- **GraphRAG** — Edge et al. [25]; **RAPTOR** — Sarthi et al. [26]. Двойной граф методологии — частный случай graph-based RAG, специализированный под код.
- **Call graph analysis в IDE** — многолетний опыт ReSharper, IntelliJ, Sourcegraph, GitHub code search.
- **CodeBERT / Codex-style модели** — Feng et al. [40], Chen et al. [41] — показывают, что модели, специализированные на коде, выигрывают от структурных признаков.

## 8. Элемент 5. Двойное назначение разметки

### 8.1. Описание

Якоря — одновременно **шаблон для генерации** и **точка разреза для селективной навигации**. При генерации агент получает контракт и пишет код внутри якорей. При поддержке — *специализированный* инструмент, осведомлённый о конвенции (в нашей реализации — команда `file_show --contracts --blocks`), позволяет селективно запросить куски файла: «покажи контракты всех функций в файле X», «покажи блок DECODE_JWT в функции validateUser». Стандартные RAG-сплиттеры (fixed-size, recursive-character, AST-based) якоря не распознают; для их интеграции в общий RAG-пайплайн требуется свой сплиттер, явно ищущий `START_*` / `END_*`.

### 8.2. Механизм действия

1. **Два режима — одна инвестиция.** Без разметки генерация и навигация — разные проблемы. С разметкой — одно и то же явление видит обе задачи.

2. **Снижает контекстную перегрузку.** Вместо загрузки всего файла на 500 строк агент получает контракты + интересующие блоки, что умещается в 1–2K токенов.

3. **Compatibility between humans and machines.** Якоря читаются и человеком, и парсером.

### 8.3. Условия применимости

Применимо всегда, когда применён элемент 3.

### 8.4. Границы и провалы

1. **Инструментарий чтения якорей.** Без команд `file_show --contracts --blocks` двойное назначение реализуется только «вручную» — агент grep'ает якоря. Это работает, но медленно и шумно.

2. **Синхронизация якорей с кодом.** Если `START_BLOCK_DECODE_JWT` остался, а реальная логика переместилась, навигация даст неверный фрагмент. Митигация — post-commit linter, проверяющий корректность блоков.

### 8.5. Связанные работы

Концепция параллельного использования одного артефакта и для генерации, и для навигации близка к подходу DSPy [39]: signatures — одновременно спецификация и runtime-подсказка. В коде аналогом являются docstring'и с machine-readable секциями (Google style, NumPy style, reStructuredText).

## 9. Элемент 6. Пропорциональная гранулярность

### 9.1. Описание

Глубина разметки и объём логирования *пропорциональны критичности* модуля:

| Класс модуля | Разметка | Логирование |
|--------------|----------|-------------|
| Критический (payment, auth, data-integrity) | Полный контракт + блоки, runtime-контракты обязательны | DEBUG + belief_state |
| Стандартный | Контракт + ключевые блоки | INFO |
| Вспомогательный (helpers) | Минимальный контракт | WARN+ |

Класс фиксируется в плане разработки через атрибут `criticality`. Инструменты анализа используют этот класс для расчёта risk-level при impact-анализе.

### 9.2. Механизм действия

1. **Избегает over-engineering.** Классификация как `helper` на три строки не требует runtime-валидации.

2. **Гарантирует покрытие критического пути.** Например Payment-модуль без DEBUG-логов и без runtime-валидации — источник продакшн-инцидентов.

3. **Сигнал для impact-анализа.** `criticality = critical` → `risk.level` повышается автоматически. Это делает предупреждения о блэст-радиусе более точными.

### 9.3. Условия применимости

Применимо всегда; только классификация требует осознанного решения.

### 9.4. Границы и провалы

1. **Неинформативная классификация.** Ценность `criticality` как атрибута — в *различении*: tooling использует её для расчёта risk-level в impact-анализе, для принятия решения о runtime-контрактах на границе, для дифференцированной глубины belief-логирования. Если **все модули** попадают в один класс — сигнал теряется и атрибут перестаёт нести информацию для инструментов. Это происходит двумя путями: (а) рефлекторная классификация «better safe than sorry», при которой всё помечается `critical` без анализа последствий; (б) поверхностная классификация «всё — helper», лишающая критический путь защиты.

   Распределение классов — **свойство домена**, не норматив. В типовой бизнес-CRUD-системе разумно (по аргументу: большинство потоков не касается денег/auth/PII) ожидать перевес `standard` модулей, меньшинство `critical` (auth, payment, data-integrity) и отдельные helper'ы. В высоконагруженных платёжных системах, системах управления медицинским оборудованием, авиационных ПО, криптографических компонентах или системах с safety-critical классификацией доля `critical` может легитимно достигать 60–80% и выше — и это корректное отражение реальности, а не ошибка классификации. Числовой порог в методологии *не фиксируется* — он невозможен без знания домена. Эти утверждения — аргументы от домена, не данные эмпирического survey.

   Критерии корректной классификации, не зависящие от домена:
   - решение о `criticality` принимается на основе конкретных последствий отказа (финансовый ущерб, безопасность, нарушение compliance, data loss) — а не «на всякий случай»;
   - в проекте присутствует **минимум два класса** на уровне M-* (если всё размечено одним классом — атрибут не работает как сигнал для tooling);
   - классификация обсуждается на уровне approval-checkpoint'а, а не присваивается ИИ-ассистентом по умолчанию;
   - пересмотр класса — часть refresh-цикла (модуль, который принимал на себя PII, из `standard` переходит в `critical`; deprecation критического UC может снизить класс потребителей).

   Провал этого элемента — не «слишком много критического», а **classification without signal**: классы распределены, но не отражают различий в реальной зоне риска.

2. **Драйф классификации.** Модуль, написанный как helper, со временем становится critical (например на него повесили обработку PII). Политика — периодический ревью классификации через `doubled-graph reviewer`.

### 9.5. Связанные работы

Принцип пропорциональной защиты — классика в reliability engineering (Beyer et al. [42] — Google SRE). В LLM-инженерии аналог — дифференцированная верификация (Dhuliawala et al. [18]; индустриальные гайды по guardrails).

## 10. Элемент 7. Код — живой документ

### 10.1. Описание

Изменение кода или контракта *автоматически* триггерит обновление связанного артефакта. Реализуется через:

- `refresh` — команду, сверяющую declared graph с кодом и предлагающую патчи;
- `analyze` — инкрементальный пересчёт observed graph после коммита;
- **git post-commit hook** как базовый механизм (работает в любом IDE);
- **IDE-specific hooks** (PostToolUse в Claude Code, custom commands в Cursor/Continue) — как ускоритель.

Если синхронизация не происходит — методология нарушена (молчаливый дрейф).

### 10.2. Механизм действия

1. **Сокращает stale-окно.** От коммита до актуального графа — секунды, не дни.

2. **Удерживает «цену правды» низкой.** Если обновление артефакта автоматическое, команда не накапливает drift-долг.

3. **Осязаемый сигнал деградации.** Если hook падает — видно сразу; если артефакт подсвечивается как stale — видно сразу.

### 10.3. Условия применимости

Применимо в любом git-репо. Требует согласия команды на post-commit автоматизацию.

### 10.4. Границы и провалы

1. **Hook на больших монорепо.** Полный `analyze` может занимать минуты. Митигация — `analyze --mode incremental --paths <touched>`.

2. **Блокировка коммита.** Если hook блокирующий и падает, разработчик не может закоммитить hotfix. Митигация — non-blocking hook: сообщает, но не блокирует (блокировку вынести в CI pre-push / PR check).

3. **Командная дисциплина.** Если часть команды отключает hooks локально, artifact/code divergence появляется на merge. Митигация — git alias в setup, CI проверка целостности на PR.

### 10.5. Связанные работы

Концепция «сode as single source of truth» встречается в области model-driven engineering и round-trip engineering (Sendall & Kozaczynski [43]). Отличие методологии — в асимметричной политике: в режиме `migration` истина — в коде, в `post_migration` — в замысле (раздел 13).

## 11. Элемент 8. Наблюдаемое belief-состояние

### 11.1. Описание

В коде, генерируемом по методологии, в ключевых точках выполнения размещаются структурированные лог-события с обязательным полем `belief_state` — короткой (≤ 200 токенов) вербализацией **runtime-допущения**, которое ИИ-ассистент материализовал при написании этой функции. Формат события:

```json
{
  "ts": "2026-04-18T10:22:04.123Z",
  "level": "info",
  "anchor": "validateUser:BLOCK_DECODE_JWT",
  "module": "M-AUTH-VALIDATE",
  "requirement": "UC-001",
  "correlation_id": "req_abc123",
  "event": "jwt_decoded",
  "belief": "payload trusted, clock-skew tolerance 60s"
}
```

Важно различать два явления. **Belief-событие — это не трассировка внутреннего рассуждения LLM** (оно структурно недоступно без доступа к весам модели — Elhage et al. [46], Meng et al. [47]) и не chain-of-thought-трейс модели, который к тому же может быть unfaithful (Turpin et al. [5], Lanham et al. [6]). Это **зафиксированное в коде** утверждение о runtime-состоянии в конкретной точке, эмитируемое приложением во время исполнения и проверяемое внешним наблюдателем по факту.

**Кто записывает belief в код:**

1. **ИИ-ассистент во время генерации** — по промпту, предписывающему встраивать `log.info(...)` с полями `anchor`, `module_id`, `requirement_id`, `belief`, `correlation_id` на границах модуля и в критических блоках.
2. **Человек при ручной правке** — по конвенции, зафиксированной в `AGENTS.md` и language-адаптерах методологии.

**Автоматической инструментации «сверху» нет.** Инструмент `doubled-graph` сам belief-логи не вставляет и не генерирует; linter проверяет только корректность цепочки ID и парность якорей, но не наличие `belief`-поля в каждом лог-вызове. Методология задаёт *конвенцию* и *схему поля*, а не runtime-enforced контракт.

**Два уровня belief — статический и рантайм-.** Механизм работает на стыке двух разных артефактов, и их важно не путать:

1. **В коде (статически).** ИИ-ассистент при генерации пишет вызов logger'а с захардкоженной `belief`-строкой:
   ```python
   log.info(
       "jwt_decoded",
       extra={
           "anchor": "validateUser:BLOCK_DECODE_JWT",
           "belief": "payload trusted, clock-skew tolerance 60s",
       },
   )
   ```
   Строка «payload trusted, clock-skew tolerance 60s» — литерал исходного кода, материализованное *в момент написания функции* допущение ИИ-ассистента. Оно попадает в git и живёт как любой другой код. Статическая часть поля `belief` — `event`, `anchor`, `module`, `requirement` — тоже вшивается в код.

2. **В рантайме (динамически).** Когда функция исполняется, logger эмитирует JSON-событие, в котором статические поля соседствуют с динамическими (`ts` — из системного времени, `correlation_id` — из request-контекста, конкретные значения переменных, если их решено включить). Это событие и есть то, что потребляется observability-стеком.

То есть один и тот же `belief` существует в двух формах: как литерал в git и как поле JSON-события в логе. Первый — артефакт генерации, второй — артефакт исполнения; первый обязателен для второго; второй невозможен без первого. Если говорить о «belief-логе», обычно имеется в виду второй, но его ценность целиком зависит от того, что и как написано в первом.

Наблюдаемое belief-состояние позволяет задним числом отличать «функция вернула неверный результат, потому что неверно интерпретировала вход» от «функция вернула результат, корректный для своего допущения, а ошибочным оказалось само допущение».

### 11.2. Механизм действия

Важно предварительно разделить, **на какой вопрос belief-лог отвечает и на какой — нет**.

| Вопрос | Отвечает ли belief-лог? |
|---|---|
| Какое допущение было зашито автором в этот участок кода? | Да (литерал в коде) |
| Совпало ли это допущение с фактическим runtime-состоянием? | Да (лог-событие рядом с фактическими значениями из контекста) |
| Где именно произошло отклонение реальности от допущения? | Да (якорь привязывает событие к конкретной точке) |
| Почему ассистент в конкретной сессии сгенерировал именно такой код? | **Нет.** Это другой слой наблюдаемости (см. § 2.7) — инференс-трассировка ассистента (LangSmith, Phoenix и т.п.), требующая собственной инфраструктуры |
| Что LLM «думала» на inference? | **Нет.** Структурно недоступно без доступа к весам (Elhage et al. [46], Meng et al. [47]) |

С учётом этого разделения — три конкретных механизма действия:

1. **Обходит faithfulness-разрыв в отношении *исполненного кода*.** CoT-трейс внутри модели может быть отделён от реального вычисления (Turpin et al. [5]). Belief-событие, эмитируемое сгенерированным кодом во время исполнения, *не* пытается воспроизвести внутреннее рассуждение LLM — оно фиксирует **зашитое в код допущение** и время, когда это допущение встретилось с реальными данными. Это обходит faithfulness-разрыв не через попытку прочитать мысли модели, а через перенос верифицируемого утверждения в исполняемый артефакт.

2. **Делает возможной отладку постфактум в коде.** При инциденте без belief-логов команда знает, что функция вернула неверный результат, но не знает, *какое допущение было некорректным*. Belief — именно такая материализация, привязанная к якорю, модулю и требованию через элемент 9. Важно: этот механизм отвечает на «почему *эта функция* повела себя так?», но не на «почему ассистент *написал эту функцию* именно так» — второй вопрос требует LLM-слоя observability (§ 2.7, слой 1).

3. **Позволяет измерять калибровку допущений в коде.** Сопоставление зафиксированных belief со фактическими исходами — прямой путь к агрегированной оценке, в каких классах ситуаций допущения, материализованные ассистентом в коде, систематически верны, а в каких — систематически ошибочны. Это не оценка «качества модели», это оценка «качества кода, который модель генерирует» — прикладная и измеряемая без доступа к инференсу.

### 11.3. Условия применимости

Элемент имеет смысл, когда выполнены оба уровня:

**На уровне кода:**
- ИИ-ассистент подчиняется промпт-конвенции методологии и встраивает в генерируемый код вызовы logger'а с `belief`-полем (в автономных сценариях или за пределами методологии belief не появится сам собой);
- соблюдается пропорциональная гранулярность (раздел 9) — belief в критических модулях обязателен, в helper'ах не нужен.

**На уровне рантайма:**
- в проекте есть structured-logging-стек (Python `structlog`, TS `pino`, Go `slog`, Java SLF4J с JSON-encoder) — без него `extra={"belief": "..."}` превращается в часть текста сообщения вместо отдельного поля, и аналитика не может отфильтровать по нему;
- развёрнут какой-либо агрегатор логов или система трассировки (Loki, Grafana Tempo, Langfuse, Phoenix, обычный ELK) для анализа задним числом; базовая конфигурация методологии не требует конкретного выбора, но без чего-либо из этого JSON-события оседают в файлах, которые никто не читает.

### 11.4. Границы и провалы

1. **Элемент держится на дисциплине, не на tooling.** Linter не ловит отсутствие `belief`-поля. При правках через grep-and-replace или через внешний агент, не знающий методологию, belief-строки могут тихо пропадать. Митигация — включение проверки наличия `belief` в критических модулях в CI-линт (proposed-расширение, в текущей реализации отсутствует).

2. **Verbosity и стоимость.** Логирование belief на каждый вызов раздувает объём логов и повышает стоимость хранения/обработки. Митигация — пропорциональная гранулярность (раздел 9): belief-логирование обязательно только в критических модулях, опционально в стандартных, отсутствует в helper-модулях.

3. **Самообман.** ИИ-ассистент может написать `belief`-строку, не соответствующую семантике сгенерированного кода (аналогично unfaithful CoT). Это не устраняется самим логированием — оно устраняется внешними проверками: runtime-контрактами на границе модуля, тестами, Chain-of-Verification на этапе ревью.

4. **Privacy.** В `belief`-строке, написанной ИИ, может случайно оказаться PII из контекста (например, конкретные email-ы или номера карт, если они были видны при генерации). Митигация — post-processing лога с маскированием PII; запрет на включение сырых значений переменных в belief-строку — только абстрактные характеристики состояния.

5. **Инструментация «сверху» недоступна.** Если в проекте исторически нет structured logging, ретрофитить belief-события во все критические точки — отдельная работа, которую не решает ни методология, ни инструмент. В режиме `migration` это часть прохода по bounded context.

### 11.5. Связанные работы

- **Структурированное логирование и distributed tracing** как общая инфраструктурная область — Sigelman et al. [14] (Dapper, Google, 2010) и стандарт OpenTelemetry [45]. Эти работы описывают *инфраструктуру наблюдаемости* (span/trace/correlation_id, семантика атрибутов) — ту техническую основу, на которой практика LDD реализуется в рантайме. Они *не* являются методологическим источником LDD: Dapper решает задачу трассировки запросов между сервисами, не задачу фиксации авторских допущений в коде, сгенерированном LLM. Концепция belief-логирования как инженерной практики LLM-разработки не имеет единого академического источника — она сложилась в индустриальной работе с LLM-observability инструментами (LangSmith, Arize Phoenix, W&B Weave, OpenLLMetry, Langfuse) и в дискуссиях prompt-engineering сообщества. Настоящая работа использует этот материал как прикладной, не как established research-field с устоявшейся таксономией; авторитет Dapper/OpenTelemetry распространяется только на технический слой, на котором LDD исполняется, а не на саму методику.
- **LLM observability** — современные инструменты (LangSmith, Phoenix, W&B Weave, OpenLLMetry) предоставляют tracing для LLM-вызовов.
- **Interpretability** — работы по механистической интерпретации (Elhage et al. [46]; Meng et al. [47]) дают долгосрочный ответ на «что модель *реально* думает». Belief-логирование — практическое приближение, доступное уже сегодня.

## 12. Элемент 9. Сквозная прослеживаемость

### 12.1. Описание

Прослеживаемость связывает четыре уровня: бизнес-намерение (`UC-*` в `requirements.xml`), архитектурное решение (`M-*` в `development-plan.xml` и `knowledge-graph.xml`), приёмочные проверки (`V-M-*` в `verification-plan.xml`), и исполняемый код с лог-событиями (якоря + `log.*(... anchor, module_id, requirement_id ...)`).

**Структура связи — граф M:N, не линейная цепочка.** Это принципиально: любая попытка свести её к «UC → M → V-M → anchor → log» справедлива только для простейшего случая 1:1:1 и не описывает реальные репозитории.

**Реальные кардинальности:**

| Связь | Кардинальность | Пример |
|---|---|---|
| UC → M | 1:N | UC-001 «login» реализуется модулями `M-AUTH-VALIDATE` + `M-SESSION-CREATE` + `M-RATE-LIMIT` |
| M → UC | M:N | `M-DB` (shared database client) обслуживает UC-001, UC-042, UC-118 |
| M → V-M-* | 1:N | один модуль имеет несколько тестов (unit/integration/e2e) |
| anchor (функция/блок) → UC, M, V-M-* | M:N | функция `sanitize_input` может ссылаться на UC-042 и UC-118 одновременно |

**Механика связи в коде и артефактах.** Множественность реализуется через **LINKS — comma-separated список ID** в комментарии MODULE_CONTRACT и в function-level контрактах:

```ts
// START_CONTRACT: sanitize_input
// PURPOSE: strip control characters from user input
// LINKS: UC-042, UC-118, M-INPUT-VALIDATION, V-M-INPUT-VALIDATION-01
// END_CONTRACT: sanitize_input
```

В графе знаний (`knowledge-graph.xml`) связи между модулями отражает `<CrossLink from="M-X" to="M-Y" relation="uses" />` — это **M→M**, не M→UC. Прямой M→UC связи в XML-артефактах upstream нет; привязка модуля к use case'ам выражается только через поле LINKS в MODULE_CONTRACT кода. В `verification-plan.xml` атрибут `USE_CASES="UC-001,UC-042"` у `<VF-*>` verification flow'ов формально допускает список — это используется для сценариев, покрывающих несколько UC.

**Типы модулей в графе знаний.** Upstream knowledge-graph поддерживает типы `ENTRY_POINT`, `CORE_LOGIC`, `DATA_LAYER`, `UI_COMPONENT`, `UTILITY`, `INTEGRATION`. Модули типа `UTILITY` (shared logger, db client, http client) по природе обслуживают многие UC и получают **косвенную** привязку к бизнес-намерению через CrossLinks от потребителей: цепочка `UC-042 → M-CHECKOUT → (CrossLink uses) M-DB`. Прямое поле LINKS в MODULE_CONTRACT у utility-модуля может быть пустым в части `UC-*` — это норма, пока на него ссылается хотя бы один модуль с UC.

Разница orphan-модуля и shared-модуля:

- **Shared-модуль** — не имеет прямых `UC-*` в LINKS, но имеет входящие CrossLinks от модулей, у которых есть UC. Валиден.
- **Orphan-модуль** — не имеет ни `UC-*` в LINKS, ни входящих CrossLinks. Сигнал к удалению или перепривязке.

### 12.2. Механизм действия

1. **Привязывает техническое к бизнесу.** Когда lead говорит «мы больше не обрабатываем P2P-переводы», поиск по UC-ID в LINKS комментариях MODULE_CONTRACT и по `USE_CASES` атрибутам VF-элементов даёт список затронутого кода и верификационных сценариев.

2. **Снижает «код неизвестного происхождения».** В долгоживущих репозиториях (5+ лет, смена команды или архитектора) многие инженеры сообщают о модулях, которые *работают*, но никто в текущем составе команды не знает, зачем они там — классический risk-паттерн, зафиксированный в литературе по software maintenance (см., напр., Binkley [15], Storey [16]). Настоящая методология запрещает orphan-модули (без UC и без входящих CrossLinks). Shared-модули разрешены — но связь с бизнес-намерением выводится транзитивно через граф.

3. **Запросы по коду в логе.** Лог-событие содержит `requirement_id`, который для функций, служащих нескольким UC, может быть либо конкретным UC из request-контекста (какой use case сейчас активен), либо comma-separated списком. Методология фиксирует только схему поля — конкретная стратегия (single vs list) — инженерное решение команды.

### 12.3. Условия применимости

Элемент применим в любой кодовой базе, предполагающей поддержку дольше одного спринта. Multi-UC случай встречается в любом проекте больше пяти модулей (shared infra всегда появляется), поэтому инструментарий и разметка проектируются с учётом M:N сразу, не оставляя это «на потом».

### 12.4. Границы и провалы

1. **Прослеживаемость держится на конвенции, не на tooling.** Текущий upstream-lint проверяет только **присутствие поля LINKS** в MODULE_CONTRACT; содержимое строки не парсится, существование каждого ID в соответствующем XML-файле не верифицируется, orphan-детекция по LINKS не выполняется. Сверка «каждый `UC-*` из LINKS существует в `requirements.xml`», «модуль не-UTILITY-типа имеет хотя бы один `UC-*` в LINKS», «функция, встречающаяся в логе с `requirement_id=UC-042`, имеет `UC-042` в LINKS» — это proposed-расширение lint'а, не реализованное в текущей версии. В реальном использовании это закрывается ручным grep'ом, git hook'ами команды и дисциплиной.

2. **Перегенерация ID.** Если UC-001 был удалён и позже создан UC-001' с той же нумерацией, но иной семантикой — прослеживаемость компрометирована: старые ссылки в коде и логах ведут «в никуда» и одновременно «в что-то». Митигация — ID не переиспользуются; удаление фиксируется пометкой `deprecated` в `requirements.xml`, номера из удалённых UC в новых use case'ах не используются.

3. **Granularity mismatch.** Одно требование может реализоваться десятком модулей, один модуль — обслуживать десять требований. Это не провал методологии, а её нормальный режим (M:N — см. § 12.1). Провалом становится *игнорирование* множественности: если LINKS содержит только первый UC из набора, связь с остальными теряется. Митигация — дисциплина при генерации кода ИИ: промпт (`prompts/new-project/06-generate.md` и аналогичные в сценарии миграции) требует перечислять *все* применимые ID в LINKS.

4. **Мигрированные legacy-модули.** Для реверс-инжиниринга восстановленные UC могут быть неточны: ИИ-ассистент угадывает, для какого use case был написан данный кусок кода. Митигация — в режиме `migration` цепочка восстанавливается как «лучшая гипотеза» с обязательной human approval на уровне UC (§ 7.2 workflow); спорные случаи фиксируются в `docs/DRIFT.md` до разрешения человеком.

5. **Стратегия `requirement_id` в рантайм-логе при multi-UC.** Методология не фиксирует, как именно поле заполняется, когда функция служит нескольким UC: брать активный UC из контекста запроса, писать список или дублировать событие для каждого UC. Это — решение команды при настройке логгера; базовая конфигурация методологии не требует определённого выбора, но внутри одного проекта стратегия должна быть единой, иначе аналитика деградирует.

### 12.5. Связанные работы

Requirements traceability — классическая тема SE (Gotel & Finkelstein [48]; обзор — Cleland-Huang et al. [49]). ИИ-методология отличается тем, что прослеживаемость опирается не только на файлы артефактов, но на проверяемую цепочку *в коде* через якоря.

## 13. Элемент 10. Управляемая автономия

### 13.1. Описание

Разделение ролей:
- **Человек** — архитектор и верификатор: утверждает артефакты, принимает приёмочные gates, решает неоднозначные drift-случаи, устанавливает класс критичности модулей, переключает режим.
- **ИИ** — исполнитель в рамках каркаса: генерирует код по утверждённому плану, пишет тесты, размечает якоря, обновляет `docs/*.xml` через `refresh`, предлагает план рефакторинга.

Граница проходит по approval-checkpoint'ам (элемент 2). За пределами — агент останавливается и спрашивает.

### 13.2. Механизм действия

1. **Ограничивает damage radius.** Агент с полной автономией, ошибаясь в архитектурном решении, может за одну сессию существенно переработать кодовую базу. Gate на контракт-изменениях останавливает это.

2. **Делает ответственность определённой.** Человек отвечает за утверждённые артефакты; ИИ — за исполнение. При инциденте belief-лог в коде позволяет установить, *какое допущение в коде не сохранилось в рантайме и на каком якоре*. Отдельный вопрос — почему ассистент при генерации зафиксировал именно это допущение — требует обращения к LLM-трассировке ассистента (§ 2.7, слой 1), если она велась; сам по себе belief в коде на этот вопрос не отвечает.

3. **Сохраняет человеческое мышление.** При полной автономии инженер теряет навыки — он больше не моделирует систему в голове. При управляемой автономии одобрения требуют активного размышления — это «интеллектуальная тренировка» на критических точках.

### 13.3. Условия применимости

Всегда. В режиме `fast` (для многоагентных волн) автономия максимальна, но остаётся up-front approval плана.

### 13.4. Границы и провалы

1. **Слишком много автономии.** Сценарии без одобрений (агент сам пишет, коммитит и сливает изменения без участия человека) качественно ассоциируются с более частыми инцидентами — неподтверждённые архитектурные изменения, самоусиливающиеся ошибки на этапе цикла исправлений, коммиты с нарушенными контрактами. Точная количественная оценка разрыва («во сколько раз чаще») автору из опубликованных данных неизвестна; индустриальные отчёты и рекомендации устоявшихся практик (Anthropic «Building effective agents», OpenAI practical agent guide) согласно рекомендуют участие человека в цикле (human-in-the-loop) для высокорисковых действий — операций, необратимых в своих последствиях. Контролируемые эксперименты, сравнивающие сценарии без одобрений и сценарии с одобрениями на сопоставимой кодовой базе и наборе задач, — открытое направление исследований.

2. **Слишком мало автономии.** Если gate стоит перед каждой строкой кода — методология тормозит работу. Gate нужен перед *необратимыми* действиями, а не перед рутинными.

3. **Gate becomes rubber stamp.** Человек, устав от одобрений, начинает жать «ОК» автоматически. Это эквивалентно отсутствию gate. Митигация — структура представления решения: агент *обязан* выделить, что именно требует одобрения (чтобы увидеть это человек должен прочитать).

### 13.5. Связанные работы

Human–AI teaming — Amershi et al. [32]; Guidelines for Human-AI Interaction (Amershi et al. [32]; Microsoft HAX guidelines). Теория автоматизации — Parasuraman et al. [50] (таксономия уровней автоматизации). Методология позиционируется в диапазоне 4–6 уровней Parasuraman (system suggests a decision → checks with human → executes on approval) — это дизайн-выбор, согласованный с общей рекомендацией индустриальных гайдов для высокорисковых задач (Anthropic «Building effective agents»; OpenAI practical agent guide), но не доказанный контролируемым экспериментом в рамках настоящей работы.

## 14. Взаимодействие элементов

### 14.1. Комплементарность

Элементы методологии — не независимые улучшения; они поддерживают друг друга, но в разной степени. Честная классификация:

**Фундаментальные элементы** (удаление ломает связи между остальными):

- **Элемент 1 (замысел) ↔ Элементы 2, 3, 9.** Без формализованного замысла нечего утверждать, некуда ссылаться из якорей, не с чем строить сквозную цепочку. Якоря с UC-ссылками, указывающими на пустые требования, — бессмысленны.
- **Элемент 2 (утверждённый чертёж) ↔ Элемент 10.** Approval без объекта — формальность; автономия без границ — ничего не ограничивает. Эти два элемента тесно взаимозависимы.
- **Элемент 3 (семантический каркас) ↔ Элементы 4, 5, 9.** Без якорей observed graph можно построить через AST, но declared граф с привязкой к намерению — нельзя; двойное назначение разметки исчезает; прослеживаемость в коде теряет точку привязки.
- **Элемент 4 (двойной граф) ↔ Элемент 7.** Без двух графов живой документ не знает, *что* синхронизировать. Refresh превращается в «регенерацию из кода», а не сверку.
- **Элемент 7 (живой документ).** Без автоматической синхронизации drift накапливается молча, и весь остальной каркас деградирует на горизонте недель.

Удаление любого из этих элементов не «убавляет 10%» — оно ломает одно из рабочих свойств системы (утверждаемость, прослеживаемость, синхронность, граничность автономии).

**Элементы-усилители** (удаление снижает эффективность, но не ломает основной контур):

- **Элемент 5 (двойное назначение разметки)** — дериват элемента 3. Без него якоря остаются, но инструменты навигации типа `file_show --contracts --blocks` теряют значение. Методология работоспособна; RAG-навигация по коду становится менее удобной.
- **Элемент 6 (пропорциональная гранулярность)** — оптимизация, а не несущая часть. Без неё разметка и логирование однородны: вспомогательные модули размечены на уровне критических, или критические — поверхностно. Первое — перерасход усилий, второе — риск в критическом пути. Но методология *как таковая* продолжает работать.
- **Элемент 8 (наблюдаемое belief-состояние)** — усилитель отладки задним числом. Без него критические инциденты отлаживаются «обычным» способом — по tests, tracing, git log. Дольше и менее полно, но возможно. Большинство элементов методологии продолжают работать.
- **Элемент 9 (сквозная прослеживаемость)** — усилитель связи технического и бизнес-уровней. Без неё cleanup (что удалить при декомиссии UC), root-cause (какое требование стоит за багом) и orphan-детекция становятся задачами ручного поиска. Код остаётся размеченным, верифицируемым и управляемым; просто дороже становится бизнес-запрос «что удалить».

Удаление элемента-усилителя даёт *работоспособную деградированную* версию методологии. Это компромисс, не катастрофа. Настоящая работа рекомендует каждый усилитель как полезное расширение на основании аргументации выше — это дизайн-позиция автора, согласованная с общей направленностью методологии (управляемость, anti-drift, наблюдаемость), а не эмпирически измеренное улучшение.

**Практическое следствие.** Если команда не готова к полной методологии сразу, минимально-рабочий набор — элементы 1, 2, 3, 4, 7, 10 (шесть фундаментальных). Элементы 5, 6, 8, 9 можно добавлять итеративно, по мере появления ресурсов и понимания. Это частичная методология — не 80% эффекта полной, но и не «другая система». Она даёт основной механизм управляемости, ценой снижения эффективности отладки (без 8, 9), оптимизации усилий (без 6) и удобства навигации (без 5).

Важно: при таком частичном принятии *остаются консистентными* те элементы, которые приняты. Выборочное применение элемента 3 «только к половине модулей» ломает methodology systemically — в этом случае lint, двойной граф и любая автоматическая сверка работают частично; полупринятие хуже, чем последовательное не-принятие конкретного элемента.

### 14.2. Потоки контроля и данных

Контур методологии замкнут:

```
Замысел (1,2)
    ↓
Артефакты в XML (1,9)
    ↓
Декомпозиция на модули с criticality (1,6)
    ↓
Approval-gate (2,10)
    ↓
Кодогенерация → якоря (3,5)
    ↓
Двойной граф: declared + observed (4)
    ↓
Живой документ: hooks, refresh (7)
    ↓
Выполнение → belief-логи (8)
    ↓
Сквозная прослеживаемость (9)
    ↓
[Detect drift] → Политика (13)
    ↓
[Approval-gate (2,10)] → Коррекция → возвращаемся к артефакту или коду
```

Контур — цикл, а не водопад. После первичной генерации система живёт в режиме поддержки (раздел 16.3) — непрерывная сверка declared ↔ observed.

### 14.3. Политика дрейфа: ключевой элемент связности

Дрейф между declared и observed может разрешаться в двух направлениях:
- «Код прав» — артефакт догоняет (режим `migration`, а также `post_migration` при признании, что код реализует *новое требование*, не зафиксированное);
- «Артефакт прав» — код исправляется (режим `post_migration`, когда отклонение признано багом).

Решение не принимается ИИ самостоятельно. ИИ предлагает классификацию (опираясь на type дрейфа и историю): `bug` / `new_requirement` / `inconsistent_intent`. Человек утверждает, либо помещает случай в `DRIFT.md` (если разобрать сейчас невозможно).

Переключение между режимами — явное поле в `AGENTS.md`:

```markdown
<!-- doubled-graph:phase:start -->
## doubled-graph phase
phase: post_migration
updated: 2026-04-18
<!-- doubled-graph:phase:end -->
```

Это поле — канонический «источник правды» о текущем режиме. ИИ читает его первым при входе в репо и ведёт себя соответственно.

### 14.4. Масштабирование: что меняется с размером проекта

Методология применяется на одинаковой глубине ко всем проектам. Масштаб влияет на *количество* артефактов (3 UC vs 300), не на их *структуру*. Это осознанное решение: tooling поддерживает одну форму; «облегчённые» проекты молча ломают работу инструментов (например, отсутствие `critical-path` отметок → risk-level недооценивается → impact-анализ становится советчиком-оптимистом).

Альтернатива для очень маленьких pet-проектов — не применять методологию совсем (без гибридов «методология-лайт»).

### 14.5. Ориентация на локальные модели

Методология оптимизирована под локальные open-weights модели среднего размера (≈30–70 B плотных параметров), развёртываемые в локальной или on-prem инфраструктуре. Актуальный набор моделей этого диапазона на момент написания (2026): Qwen3-32B, Qwen2.5-72B, Qwen3-Coder-30B, Llama-3.3-70B, Llama-3.1-70B, GLM-4-32B и их ближайшие преемники.

*Что сознательно вынесено за рамки этой ориентации.* Open-weights MoE-модели с большим суммарным объёмом и малой долей активных параметров (Qwen3-235B-A22B, DeepSeek V3, Llama 4 Scout/Maverick, GLM-4.5 и т. п.) выходят за диапазон 30–70 B плотных параметров: хотя активная доля может быть сопоставима, требования к памяти развёртывания совсем другие, и у них есть frontier-подобные возможности. Методология на них применима, но настройки (длина контекста, агрессивность structured decoding, плотность externalize-state) в этом классе становятся избыточно консервативными — их разумно ослабить по месту.

Ориентация на 30–70 B плотных диктует четыре практики, от которых зависит работоспособность всего контура:

- **Компактные промпты.** Каждый промпт укладывается в ≤ 8 KB, чтобы не перегружать контекст локальной модели.
- **Structured decoding.** Там, где ответ должен быть в JSON или XML — используется grammar-guided generation (Outlines, llama.cpp grammar, xgrammar и т. п.). Для моделей без frontier-уровня reasoning это часто разница между рабочим парсером и «почти-валидным» выводом.
- **Externalize state.** Длинные диалоги хранят состояние в scratchpad'ах на диске, не полагаются на историю сессии.
- **Graph over context.** Вместо «положи 1M токенов в контекст» — «дай мне нужные 2K через граф».

На frontier-моделях закрытых провайдеров (Claude Opus, GPT-5.x, Gemini 2.x Pro и т. п.) методология тоже работает, но специально под них не оптимизируется.

## 15. Отношение к альтернативным подходам

Методология двойного графа — не единственный способ работы с ИИ-ассистентами. Ниже — сопоставление с шестью подходами, с которыми методология фактически конкурирует.

### 15.1. Сравнительная таблица

|---|---|---|---|---|---|---|
| **Trunk + Copilot / Cursor inline** (массовая практика) | ~0 | Высокая | Нет | Нет | Любой с LLM-inline | Пет-проекты, разведка, краткосрочная разработка |
| **BMAD (Agile brownfield)** | Средний (PRD, stories) | Средняя | Частично (через story-дисциплину) | Да (исторический профиль) | Любой | Enterprise с привычным Agile |
| **Full-agentic (Devin-style, AutoDev)** | Низкий | Очень высокая на типовых задачах | Частично (через test-driven verification) | Частично | Sandbox + test harness | Задачи с чётким test-criterion |
| **Spec-driven tools** (Kiro, aider architect) | Средний | Средняя | Частично (declared only) | Частично | IDE-specific | Greenfield с плановой фазой |
| **GRACE upstream (без doubled-graph)** | Высокий | Низкая | Частично (declared only) | Нет (только greenfield) | IDE с skills + MCP | База методологии без graph-слоя |
| **Formal methods (TLA+, Coq, Dafny)** | Очень высокий | Очень низкая | Да (в пределах модели) | Не применимо | Отдельный toolchain | Критические ядра (авиация, криптография) |
| **Двойной граф (настоящая работа)** | Высокий | Низкая–средняя | Да (declared↔observed) | Да, с оговоркой*: режим `migration` определён формально, но качество восстановленного declared-графа ограничено исходной документированностью legacy | MCP + tools | Поддерживаемые проекты средних+ размеров, смешанные greenfield/brownfield |

*\* Оговорка о brownfield-поддержке (скрытая неоднородность критерия).* Строка «Да (явный режим migration)» в сравнительных таблицах легко читается как симметричная greenfield: «методология одинаково работает и там, и там». Это не так. Режим `migration` гарантирует *процедуру* восстановления declared-графа из кода, но не его *качество*. Качество верхнеограничено исходной документированностью legacy: хорошо структурированный код с говорящими именами, тестами и docstring-ами даёт declared-граф, близкий к спроектированному с нуля; код без комментариев, без тестов, с перегруженными функциями и неявным состоянием — даёт declared-граф, в котором наблюдаемая сторона корректна (из AST), но декларируемая сторона — это лучшая реконструктивная гипотеза ИИ + интервью с командой. Второй случай принципиально не эквивалентен greenfield-варианту, где замысел фиксируется до первой строки кода; в сравнении с альтернативами критерий «поддержка brownfield» у всех подходов включает этот потолок, и прямое сопоставление бинарных «Да/Нет» в таблице создаёт иллюзию симметрии, которой нет. Там, где это решающий фактор выбора, команде стоит оценивать не факт существования режима, а ожидаемое качество declared-графа на её конкретной кодовой базе (плотность документации, покрытие тестами, сложность неявных зависимостей — §§ 7.4 п. 6, 11.4 п. 5).

### 15.2. Trunk + Copilot/Cursor inline — массовая практика

ИИ как inline-автокомплит и чат без upfront-плана, без якорей, без `docs/*.xml`. Так работает большинство команд в 2026 году.

**Где объективно выигрывает:** нулевые накладные расходы; скорость первой итерации; работа с любым IDE и моделью; не требует согласия команды на дисциплину артефактов.

**Где проигрывает:** на горизонте > 1 спринта дрейф замысла накапливается; на team-size > 1 разработчика согласованность интерпретации требует ad-hoc договорённостей; в критических доменах отсутствие runtime-контрактов и verification-gates приводит к инцидентам, которые методология с графом и якорями предотвращает или локализует.

**Сравнение:** для pet-проектов, MVP в режиме «выкинем через три месяца» и краткосрочного exploration trunk+Copilot объективно *лучше*. Наша методология окупается не всегда, а за определённым горизонтом поддержки и размера команды. Команда, принимающая решение «брать методологию или нет», должна оценить этот горизонт честно — не в пользу артефактов автоматически.

### 15.3. BMAD — Agile для ИИ-разработки

BMAD (Breakthrough Method for Agile AI-Driven Development) — структурированный подход на основе Agile: PRD, epic breakdown, user stories, acceptance criteria. Ориентирован на enterprise-команды с привычной Agile-культурой.

**Где выигрывает BMAD:** глубокая интеграция с PM-ритуалами; меньше требований к техническим деталям (не нужен AST, call-graph); развитая экосистема промптов для brownfield-discovery; привычность для PM/PO-команд.

**Где проигрывает:** нет двойного графа → drift обнаруживается случайно, а не автоматически; семантической разметки кода нет → impact-анализ делается вручную через чтение; для критических доменов с runtime-контрактами BMAD предлагает меньше.

**Сравнение:** BMAD и двойной граф — **комплементарные**, а не конкурирующие подходы. BMAD управляет management-слоем (что делаем и зачем); двойной граф управляет code-слоем (что зашито и согласовано ли). Упоминание BMAD в § 4.4 п. 3 как «generic architecture failure mode» касается *конкретного brownfield-паттерна*, не методологии в целом; это уточнение следует читать именно так.

### 15.4. Full-agentic autonomous coding

Devin, AutoDev, auto-commit агенты, Claude Code в agentic-режиме без human approval — ИИ работает в sandbox, пишет код, запускает тесты, открывает PR, иногда коммитит без участия человека.

**Где выигрывает:** радикально выше пропускная способность на задачах с чётким automated success criterion (всё, что выражается тестом); использует verification через execution, а не через approval.

**Где проигрывает:** на задачах, где успех определяется архитектурно (не тестом), prone к reward hacking — «тесты зелёные, задача не решена»; high blast radius при ошибке; trust-assumption на encapsulation sandbox'а.

**Сравнение:** full-agentic и наша методология — **разные позиции на спектре автономии**, не противостоящие подходы. Методология допускает движение к fast-профилю multi-agent (§ 13.4) для mature codebase со strong verification — это *локальный full-agentic внутри gated-flow*. Выбор между ними зависит от отрасли и стоимости ошибки.

### 15.5. Spec-driven tools — ближайший конкурент

Kiro (AWS), aider architect-mode, Cline plan-act, emerging AI-DLC frameworks — инструменты с явной плановой фазой перед генерацией. Концептуально близки к нашей методологии.

**Где выигрывает spec-driven:** встроены в конкретный инструмент (Kiro — AWS-native), промпты заточены под UI; ниже порог вхождения; не требует отдельной установки facade-MCP.

**Где проигрывает:** у большинства tools — только declared-граф, без observed; значит, drift обнаруживается только при merge/PR-review, не автоматически; brownfield-сценарий обычно не формализован; tool-lock-in (Kiro работает только в AWS-среде).

**Сравнение:** это **наиболее близкий конкурент** двойного графа. Различия — в акцентах: двойной граф добавляет observed-слой и формальный migration-сценарий, spec-driven фокусируется на UX и integrated-tooling. Для команды, работающей внутри одной IDE с встроенным spec-driven tool, переход на двойной граф даёт выигрыш только если они упираются в drift или brownfield; иначе — не факт.

### 15.6. GRACE upstream без doubled-graph — наша ближайшая исходная точка сравнения

Методология osovv/grace-marketplace без facade-MCP, skills + CLI.

**Что добавляет doubled-graph:** двойной граф (CGC в фоне), 4 native MCP-tools (`analyze`, `impact`, `context`, `detect_changes`), sensitivity-reduction интерфейс для LLM (15 curated tools вместо 14+ raw CGC tools + grace-CLI), единая точка для phase-policy (`migration`↔`post_migration`).

**Когда достаточно GRACE upstream:** нет требования к автоматической anti-drift проверке; нет brownfield-миграции; команда готова ручным grep'ом проверять consequences при рефакторинге.

**Когда нужен doubled-graph:** все перечисленные выше условия нарушены.

### 15.7. Formal methods — вне scope, но для полноты

TLA+, Coq, Dafny, Lean дают математические гарантии свойств. Correctly-stated, мощнее любой runtime-проверки.

**Где выигрывают:** математическая гарантия в пределах модели; независимость от LLM-ошибок.

**Где проигрывают:** применимы только точечно (обычно — критическое ядро < 10% LOC); требуют специализированной экспертизы; не интегрируются с agile-процессами.

**Комплементарность, не конкуренция.** Для hyper-critical систем методология двойного графа покрывает контекст вокруг formal-core, но не заменяет его. Это явно отмечено в § 5.10 и § 16.5 — formal methods вне scope.

### 15.8. Когда какой подход выбирать — эвристика

| Ситуация | Выбор |
|---|---|
| Pet-project, одиночный разработчик, < 3 месяцев жизни | Trunk + Copilot |
| MVP < 6 месяцев, один разработчик, greenfield | Trunk + Copilot или spec-driven tool |
| Enterprise Agile, сложившаяся культура PO/story | BMAD + spec-driven tool (доп. двойной граф для code-слоя) |
| Задачи с чётким test-criterion, масштабная автоматизация рутины | Full-agentic с sandbox |
| Поддерживаемый продукт > 1 года жизни, команда > 2 человек | **Двойной граф** |
| Brownfield-модернизация legacy-кодовой базы | **Двойной граф** (режим migration) — с пониманием потолка качества declared-графа (см. оговорку под таблицей § 15.1) |
| Критическое ядро (auth, payment, crypto, safety) | **Двойной граф** + formal methods для ядра |
| Greenfield с plan-first tool, который уже встроен в IDE | Spec-driven tool; двойной граф — если drift становится проблемой |

### 15.9. Позиционирование

Контрибьюция настоящей работы — *не* в том, что методология доминирует над альтернативами. Это неправда. Контрибьюция — в конкретной комбинации: **двойной граф (declared + observed) + явный режим migration + tool-agnostic facade + ориентация на локальные модели среднего размера**. Ни одна из существующих альтернатив не покрывает все четыре пункта одновременно. Для команд, которым важны все четыре, методология даёт замкнутый контур; для команд, которым достаточно любого одного или двух — подойдёт более лёгкая альтернатива.

*Важная оговорка о brownfield в этой формулировке.* «Поддержка brownfield» в списке выше — это наличие *процедуры* восстановления и формального переключения режима, не обещание, что реконструированный declared-граф совпадает по качеству с тем, что был бы построен с нуля. Качество реконструкции верхнеограничено документированностью исходного кода, покрытием тестами и сложностью неявных зависимостей (см. развёрнутую оговорку под таблицей § 15.1 и § 7.4 п. 6). Эта неоднородность симметрично присутствует во всех подходах, заявляющих «brownfield support», — прямо сравнивать бинарные «Да/Нет» в таблицах бесполезно без учёта качества legacy-источника.

Это не универсальный рецепт; это позиция на шкале подходов, сформулированная явно, с признанием, что большинство команд в 2026 году работает вне этой позиции и у них есть причины. Задача методологии — предоставить готовый инструментарий, когда команда дорастает до требования «контролируемый дрейф + поддержка brownfield (с учётом потолка качества реконструкции) + ориентация на локальные модели + независимость от конкретного инструмента», не раньше.

## 16. Границы и известные провалы

### 16.1. Когда методология не подходит

1. **Одноразовые скрипты.** Если код пишется на 30 минут и удаляется — накладные расходы на артефакты не окупаются.
2. **Research-код без пользователя.** Hack-артефакты, где ценность — скорость итерации, а не поддерживаемость.
3. **Проекты без LLM-инструментов.** Если вся разработка в vim без Copilot/Cursor/Claude Code — половина методологии избыточна.
4. **Языки без AST-инфраструктуры.** Для экзотических DSL наблюдаемый граф построить невозможно; методология деградирует в «только declared», что снижает её ценность.

### 16.2. Типичные провалы миграции

Следующие паттерны наблюдаются в индустриальной практике brownfield-модернизации (включая отчёты о внедрении BMAD-методологий, обсуждения в prompt-engineering сообществах, собственный опыт автора при прототипировании doubled-graph). Строгая статистика встречаемости отсутствует; ниже — качественное описание с указанием митигации, заложенной в промпты миграционного сценария:

1. **Generic architecture без привязки к реальному коду.** Митигация — обязательный discover-шаг, предшествующий формулировке требований.
2. **Spec drift между волнами.** Митигация — `refresh` в конце каждой волны, коммит.
3. **Переполнение context window.** Митигация — работа через граф, а не загрузка всех файлов.
4. **ИИ «улучшает» legacy за scope миграции.** Митигация — жёсткий запрет правки кода в `migration` без approval.
5. **Приёмка «полной миграции» без validation-gates.** Митигация — чеклист в `06-validation-gates.md`: lint чист, computed ≈ declared, все критические модули с контрактами.

### 16.3. Типичные провалы поддержки

1. **Hook disabled локально.** Решение — CI-check на PR.
2. **Drift silently accumulated.** Решение — периодический `doubled-graph detect_changes --scope full`, включаемый в недельный ритуал команды.
3. **LLM упирается в галлюцинацию контекста, даже с графом.** Решение — Chain-of-Verification на критических путях (независимый verifier проверяет изменения перед мержем).

### 16.4. Смещения оценки

LLM-as-judge [51] вводит ряд bias'ов: positional, length, self-evaluation. Методология в части evaluation-gates предписывает:
- Swap-and-compare (A/B симметрично);
- Diverse judge (разные модели как жюри);
- Structured rubrics (явные критерии, не «какой лучше»).

Также методология избегает использования SWE-bench Verified как приёмочного критерия (Jimenez et al. [52]), поскольку известна проблема контаминации [53]: модели могли видеть тестовые репозитории в тренировочных данных, что завышает метрики. Вместо этого предлагается ориентироваться на SWE-bench Pro, SWE-Rebench, LiveCodeBench — но не как пороги, а как индикаторы.

### 16.5. Открытые вопросы

1. **Автоматическая верификация cross-language проектов.** Монорепо с Python + TypeScript + Rust: call graph между языками неочевиден (через API, через IPC, через файлы). Методология частично решает это через «контракты на границе», но полной формализации нет.

2. **Компрессия истории длинных сессий.** Scratchpad-паттерн — рабочая практика, но формальная теория — какие события стоит сохранять, какие резюмировать, какие выбрасывать — не разработана.

3. **Метрики оценки самой методологии.** Как измерить, насколько «лучше» методология, чем её отсутствие? Предварительные подходы — A/B на одинаковых задачах с/без методологии, сравнение hallucination rate, post-coverage regressions. Но долгосрочный эффект (поддерживаемость через год-два) измеряется с запаздыванием.

4. **Взаимодействие с formal methods.** Формальная верификация (TLA+, Coq, Dafny) — из scope методологии. Но для критических систем вопрос комплементарности открыт.

## 17. Заключение

LLM-assisted development — не расширение классической инженерии ПО, а новая среда с собственными структурными ограничениями: галлюцинации как режим работы, faithfulness-разрыв рассуждения, деградация в длинном контексте, alignment-смещения. Попытка применить к ней классическую дисциплину «в лоб» даёт только частичный результат. Необходима методология, специфически спроектированная для этой среды.

Настоящая работа **предлагает** такую методологию, состоящую из десяти элементов — классифицированных на фундаментальные (удаление ломает контур) и усилители (удаление снижает эффективность): примат явного замысла, синтез по утверждённому чертежу, семантический каркас кода, контекст через двойной граф знаний, двойное назначение разметки, пропорциональная гранулярность, код как живой документ, наблюдаемое belief-состояние, сквозная прослеживаемость, управляемая автономия. Для каждого элемента **аргументирован** предполагаемый механизм действия с опорой на опубликованные свойства LLM-архитектур и устоявшиеся практики инженерии ПО.

Новизна работы — не в отдельных элементах (большинство из них имеет корни в предыдущих практиках: design by contract, log-driven development, graph-based program comprehension, retrieval-augmented generation, chain-of-verification), а в их *интеграции* в замкнутый контур с асимметричной политикой дрейфа. Технический дизайн-вклад — **двойной граф знаний** (declared + observed) с явной сверкой как предложенное ядро обнаружения и разрешения дрейфа; а также **политика переключения режимов** (`migration` ↔ `post_migration`), закрывающая разрыв между концепцией «чистой генерации из спецификации» и реальностью существующих кодовых баз.

Методология ориентирована на локальные open-weights модели среднего размера (≈30–70 B плотных параметров, развёртываемые локально или on-prem) и IDE с поддержкой tools + Model Context Protocol, что расширяет её применимость за пределы экосистем, зависящих от доступности frontier-моделей. Современные open-weights MoE с большим суммарным объёмом параметров сознательно оставлены за рамками оптимизационной цели: настройки методологии на них применимы, но становятся избыточно консервативными.

**Ограничения работы.** Это — design / position paper. Контролируемой количественной оценки выигрыша относительно альтернативных практик (см. § 15) в настоящей работе не проведено. Утверждения о частоте проблем, которые методология адресует, и об их митигации опираются на опубликованные свойства LLM-систем и индустриальную практику, не на собственный контролируемый эксперимент автора.

Направления дальнейшей работы: (1) контролируемая количественная оценка методологии на наборе задач, сопоставимых с альтернативными практиками (trunk+Copilot, spec-driven, full-agentic); (2) формализация метрик для отдельных элементов (например, drift rate в declared↔observed как функция частоты refresh'а); (3) распространение на cross-language монорепо; (4) интеграция с формальной верификацией для критических систем; (5) эмпирическое измерение долгосрочного эффекта (поддерживаемость на горизонте > 2 лет).

## Приложение A. Словарь терминов

- **Declared graph** — граф знаний, явно записанный человеком/ИИ в `docs/knowledge-graph.xml`. Представляет замысел.
- **Observed graph** — граф, вычисленный AST-парсером и анализатором call graph из реального кода. Представляет факт.
- **Якорь** — парный XML-style комментарий в коде, создающий attention-границу и точку разреза для специализированного инструмента навигации (`START_CONTRACT: name` / `END_CONTRACT: name`). Стандартными RAG-сплиттерами не распознаётся без кастомизации.
- **Approval-checkpoint** — точка в workflow, где ИИ-агент останавливается и ждёт явного одобрения человеком.
- **PhaseGate** — verification-gate из `verification-plan.xml` (module-gate, wave-gate, phase-gate). Не путать с approval-checkpoint.
- **Criticality** — атрибут модуля в `development-plan.xml`, определяющий глубину разметки и логирования: `critical` / `standard` / `helper`.
- **Drift** — расхождение между declared и observed, или между двумя версиями declared, или между контрактом и реализацией.
- **Mode (режим)** — значение поля `phase` в `AGENTS.md`: `migration` (код — источник истины) или `post_migration` (артефакты — источник истины).
- **Belief-state** — вербализованное состояние веры агента, записываемое как структурированный лог в момент принятия не-тривиального решения.
- **Bounded context** (заимствовано у DDD) — единица миграции: связная группа модулей, размечаемая в одну волну.

## Приложение B. Минимальный пример цепочки артефактов

Для модуля аутентификации в гипотетическом проекте.

**requirements.xml:**
```xml
<UseCase id="UC-AUTH-001">
  <Actor>User</Actor>
  <Action>login</Action>
  <Goal>получить session token за ≤ 3s</Goal>
  <NonFunctional>
    <Latency>p95 &lt; 3s</Latency>
    <Security>JWT-auth required</Security>
  </NonFunctional>
</UseCase>
```

**development-plan.xml:**
```xml
<Module id="M-AUTH-VALIDATE" criticality="critical">
  <Purpose>Validate JWT, resolve User.</Purpose>
  <Paths><Path>src/auth/validate.ts</Path></Paths>
  <Contract>
    <PublicExports><Export name="validateUser" /></PublicExports>
    <Preconditions><Pre>JWT header present</Pre></Preconditions>
    <Postconditions><Post>returns User | null (never throws)</Post></Postconditions>
    <Invariants><Inv>clock-skew tolerance = 60s</Inv></Invariants>
  </Contract>
</Module>
```

**verification-plan.xml:**
```xml
<Verification id="V-M-AUTH-VALIDATE-01" module="M-AUTH-VALIDATE" kind="unit">
  <Command>pnpm test auth/validate</Command>
  <Scenario>valid JWT returns User</Scenario>
  <Markers><Marker>VALIDATE_SUCCESS</Marker><Marker>VALIDATE_FAIL</Marker></Markers>
</Verification>
```

**src/auth/validate.ts** (с разметкой):
```ts
// START_MODULE_CONTRACT
// Module: M-AUTH-VALIDATE
// Purpose: validate JWT and resolve User
// Criticality: critical
// LINKS: UC-AUTH-001, V-M-AUTH-VALIDATE-01
// END_MODULE_CONTRACT

// START_CONTRACT: validateUser
// PURPOSE: validate JWT, resolve user
// INPUTS: req: Request (with Authorization: Bearer header)
// OUTPUTS: User | null (null if invalid, never throws)
// LINKS: UC-AUTH-001, M-AUTH-VALIDATE, V-M-AUTH-VALIDATE-01
// END_CONTRACT: validateUser
export function validateUser(req: Request): User | null {
  // START_BLOCK_DECODE_JWT
  const token = extractToken(req);
  const payload = decodeJwt(token);
  // END_BLOCK_DECODE_JWT

  // START_BLOCK_CHECK_CLOCK_SKEW
  if (Math.abs(Date.now() - payload.iat * 1000) > 60_000) return null;
  // END_BLOCK_CHECK_CLOCK_SKEW

  return payload.user;
}
```

**лог-событие:**
```json
{
  "ts": "2026-04-18T10:22:04.123Z",
  "level": "info",
  "anchor": "validateUser:BLOCK_DECODE_JWT",
  "module": "M-AUTH-VALIDATE",
  "requirement": "UC-AUTH-001",
  "correlation_id": "req_abc123",
  "event": "jwt_decoded",
  "belief": "payload trusted, skew=0"
}
```

Цепочка: `UC-AUTH-001` → `M-AUTH-VALIDATE` → `V-M-AUTH-VALIDATE-01` → якорь `validateUser:BLOCK_DECODE_JWT` → лог-событие. Любое её нарушение улавливается lint'ом.
